[#assertion-surface.contract]
== 断言表面

符合性断言是本书用于判定实现行为的黑盒条目。每条断言描述一个可观察场景：给定某个前置状态，用户执行某个动作，系统应出现某个可观察结果。

断言不检查内部实现。它不要求评审者知道模型事件、媒体协议、线程状态、采样频率或供应商接口。断言只检查用户或外部观察者能看到、听到或记录到的结果。

[#assertion-heading-form.contract]
=== 断言标题

每条断言使用小一级标题表达。标题前的 attrlist 提供稳定 ID、role 和规范强度。标题文本是读者可见的断言名称。

[source,asciidoc]
----
[#camera-capture-state-visible.assertion, normative=must]
=== 摄像头采集状态可见
----

`#camera-capture-state-visible` 是交叉引用地址。`.assertion` 表示标题身份。`normative=must` 表示规范强度。其他章节引用该断言时，使用 `xref:camera-capture-state-visible[摄像头采集状态可见]`。

断言不使用孤立字面量编号作为主要引用对象。编号字符串容易脱离书稿结构；标题 ID 能与目录、交叉引用和投影工具共同工作。

[#assertion-body-form.contract]
=== 断言正文

断言正文使用稳定字段描述场景。字段包括前置状态、用户动作、期望结果、不符合表现和覆盖对象。

前置状态描述用户和系统在动作发生前可观察到的状态。用户动作描述用户做了什么。期望结果描述系统应当出现的可观察结果。不符合表现列出会导致断言失败的典型行为。覆盖对象通过 xref 指向被验证的要求、流程或公共投影。

字段服务断言，不服务装饰。字段不应写内部实现细节，也不应写无法观察的心理状态。例如“模型内部收到事件”不是黑盒期望结果；“用户界面显示正在听，AI 随后回应基于用户语音”才是可观察结果。

[#assertion-strength.contract]
=== 断言强度

断言强度来自规范性语言。`normative=must` 的断言验证核心要求。失败时，实现不能核心符合。`normative=should` 的断言验证强要求。失败时，实现需要偏离声明。`normative=may` 的断言验证可选或扩展能力。实现未声明该能力时，不适用。

断言强度不是测试优先级。一个 `must` 断言即使执行简单，也比一个复杂的 `may` 断言更重要。符合性先看核心条件，再看扩展能力。

[#assertion-observation.contract]
=== 观察原则

断言以外部观察为准。观察可以来自用户界面、可听语音、字幕、状态指示、错误提示、会话记录、屏幕录制或测试工具记录。观察必须映射到用户可依赖的行为。

内部日志可以作为辅助证据，但不能替代用户可观察结果。系统内部记录“已打开摄像头”而用户界面无采集状态，不满足采集状态断言。内部记录“已取消回答”而用户仍听到旧回答继续播放，不满足打断断言。

[#assertion-scope.contract]
=== 适用范围

断言适用于实现声明的范围。核心断言适用于所有核心符合声明。条件断言适用于实现声明的条件能力。扩展断言适用于实现声明的扩展能力。

实现可以声明条件范围，但条件范围必须清楚。例如只支持移动端摄像头的实现，不接受桌面摄像头断言；但它仍必须通过移动端核心断言。声明条件范围不能取消核心构成性条件。

[#assertion-failure.contract]
=== 断言失败

断言失败表示实现没有满足该场景的可观察结果。失败可以来自功能缺失、状态冲突、错误沉默、输出错位、权限误导、上下文伪装或打断失效。

断言失败应被记录为不符合、条件限制或偏离声明。失败不能被演示成功案例抵消。一个实现偶尔成功回答视觉问题，不代表它通过所有视觉指代断言。
