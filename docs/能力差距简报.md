## 1. 报告的核心结论

报告确认：Vision Agents 当前 Qwen adapter 只覆盖了 Qwen Realtime WebSocket VAD 语音对话的一条基础路径。它能做的事情主要是：建立 DashScope WebSocket，发送 `session.update`，把输入音频重采样后发 `input_audio_buffer.append`，把视频帧转 JPEG 后发 `input_image_buffer.append`，接收 `response.audio.delta`，接收助手 transcript delta，接收用户最终转写，在 `input_audio_buffer.speech_started` 且 adapter 认为自己正在响应时发送 `response.cancel`。

这不是 Qwen3.5-Omni-Flash-Realtime 的完整适配。缺口不是一个局部 bug，而是多个官方能力对象没有在 Vision Agents adapter 中取得公共契约位置。报告把主要缺口归纳为七个对象层面：Qwen 原生 WebRTC 接入、semantic VAD、Manual WebSocket 模式、完整响应生命周期事件、完整打断投射、音频与图片输入时序状态、搜索与 Function Calling 能力面。

这个结论与我们之前对“zipball 参考价值很低”的判断一致，但报告把对象边界说得更准确：zipball 可以作为历史客户端控制模式参考，尤其是它收到 `speech_started` 后清浏览器音频队列这一点；但它不是 Qwen3.5 能力 contract，不是 Qwen 原生 WebRTC，不应该作为完整适配目标。

## 2. 官方 contract 被报告明确拆开了

报告把官方 Qwen3.5 Omni Realtime contract 拆成了独立能力项，而不是笼统说“实时全模态”。它确认了这些官方能力面：

- 模型 code：`qwen3.5-omni-flash-realtime`，控制台快照 `qwen3.5-omni-flash-realtime-2026-03-15`。
- WebSocket 连接：`wss://.../api-ws/v1/realtime?model=...` + Bearer。
- WebRTC 连接：SDP POST 到 `/api/v1/webrtc/realtime?model=...`，媒体走 RTP，事件走 DataChannel，且白名单开放。
- `session.update`：`modalities`、`voice`、`input_audio_format`、`output_audio_format`、`instructions`、`turn_detection` 等。
- VAD：`server_vad` 与 `semantic_vad`，且文档对 Qwen3.5 推荐 `semantic_vad`。
- Manual：WebSocket 可用，`turn_detection=null`，需要 `input_audio_buffer.commit` 与 `response.create`。
- 输入 transport：WebSocket 用 `input_audio_buffer.append` / `input_image_buffer.append`；WebRTC 用音频轨道和视频轨道。
- 输出 transport：WebSocket 音频走 `response.audio.delta` / `response.audio.done`；WebRTC 音频走 RTP。
- 转写事件：用户 transcription delta/completed，助手 audio transcript delta/done。
- 响应生命周期：`response.created`、`response.output_item.added`、`conversation.item.created`、`response.content_part.*`、`response.audio.*`、`response.output_item.done`、`response.done`。
- 搜索：`enable_search`、`search_options.enable_source`、`response.done.response.usage.plugins.search`。
- Function Calling：官方说明支持，但本地文档没有展开 realtime tool event schema。
- 语音控制：通过语音控制音量、语速、情绪。
- 限制：搜索与工具调用不兼容；单次会话 120 分钟；flash 音频最大 80 轮/480 秒，视频最大 50 轮/120 秒。

这个拆法对后续实现有用，因为它把“完整适配”变成了可审查矩阵，不再是泛化愿望。

## 3. Vision Agents 当前缺口

报告中最重要的是 coverage matrix。它没有把“缺口”写成抽象抱怨，而是逐项写了官方能力、官方 API surface、协议范围、模型侧职责、adapter/runtime 职责、Vision Agents 当前覆盖、状态、证据、需要补充的适配对象和验证面。

最高影响缺口是这五个：

1. **Qwen 原生 WebRTC transport 缺失**

Vision Agents 当前 Qwen adapter 只实现 DashScope WebSocket client。它不实现官方 Qwen WebRTC 的 SDP 交换、RTP 音频输入、RTP 视频输入、RTP 音频输出、DataChannel 控制事件。这一点必须和 GetStream WebRTC 分开。TideSync 当前可以通过 GetStream 做通话媒体，但这不等于已经适配 Qwen 官方 WebRTC API。

2. **semantic VAD 缺失**

官方 Qwen3.5 文档推荐 `semantic_vad`，Vision Agents adapter 固定写死 `server_vad`。这不是参数小问题，而是 Qwen3.5 Realtime 交互能力的一项官方差异没有进入 adapter contract。

3. **Manual WebSocket 模式缺失**

官方 WebSocket 支持 `turn_detection=null`、`input_audio_buffer.commit`、`response.create`。Vision Agents client 有 `commit_audio` sender，但 provider 层不暴露 Manual 模式，也没有按轮次执行 commit/create flow。

4. **完整打断投射缺失**

adapter 只在特定条件下发 `response.cancel`，没有触发 Vision Agents core 的 audio output flush，没有 emit `RealtimeAudioOutputDone(interrupted=True)`，没有丢弃 cancel 后可能迟到的旧 response delta。报告同时确认 Vision Agents core 本身已有 `RealtimeAudioOutputDone`、`RealtimeInferenceFlow.interrupt()`、`AudioOutputStream.flush()` 和 GetStream audio track flush 机制；缺口在 Qwen adapter 没有使用这些抽象。

5. **Search 与 Function Calling 缺失**

官方 Qwen3.5 Realtime 支持联网搜索和 Function Calling，且搜索与工具调用不兼容。adapter 没有 `enable_search` / `search_options` 配置，没有读取 `usage.plugins.search`，没有工具 schema 注册、tool-call event、工具执行、tool result return。

## 4. Vision Agents core 已经有部分承载结构

报告有一个关键事实：Vision Agents core 不是什么都没有。core 已经定义了这些事件对象：

```text
RealtimeAudioOutput
RealtimeAudioOutputDone
RealtimeUserTranscript
RealtimeAgentTranscript
RealtimeUserSpeechStarted
RealtimeUserSpeechEnded
RealtimeAgentSpeechStarted
RealtimeAgentSpeechEnded
```

`RealtimeInferenceFlow` 会消费这些事件。`RealtimeAudioOutputDone(interrupted=True)` 会触发 `interrupt()`；`interrupt()` 会清 transcript，清 audio output stream，并最终通过 Agent audio producer 调用 GetStream `AudioStreamTrack.flush()`。这说明本地播放清理的基础设施存在。

所以后续适配的对象不是“从零做一个 runtime”，而是把 Qwen 官方事件完整映射到 Vision Agents 已有 realtime core，并补上 core 没有表达的 Qwen3.5 官方能力对象。这个判断比之前“Vision Agents 是不是 runtime”的泛泛讨论更有用，因为它直接指出：哪些缺口是 adapter 未使用 core，哪些缺口是 core/adapter 都没有官方能力面。

## 5. TideSync 当前使用状态

报告确认 TideSync 当前默认：

```text
model = qwen3.5-omni-flash-realtime
base_url = wss://dashscope.aliyuncs.com/api-ws/v1/realtime
voice = Tina
fps = 1
include_video = True
```

TideSync 只暴露 `model/base_url/voice/fps`。它没有暴露 Qwen3.5 的 VAD 类型、Manual、搜索、工具、usage、语音控制、transcription model、音频格式等配置面。TideSync 依赖的是 `vision-agents[qwen,getstream]>=0.6.4,<0.7`，不是本地 fork。

这个事实支撑你的判断：如果要完整适配，可以 fork Vision Agents 进行魔改；当前 TideSync 没有自己的 Qwen adapter 实现，能力缺口主要在 Vision Agents Qwen plugin 和它与 core 的映射上。

## 6. 报告指出的 live API 未知项

报告没有把不能从文档/源码确认的内容伪装成事实。它列出了需要真实 API 验证的未知项：

- 裸 WebSocket 是否接受 `pcm16` / `pcm24`，还是必须严格使用文档中的 `pcm`。
- `gummy-realtime-v1` 是否仍适用于 Qwen3.5 realtime；官方示例更偏 `qwen3-asr-flash-realtime`。
- Function Calling realtime 事件 schema 与 tool result return contract。
- `semantic_vad` 的真实事件顺序和 barge-in 行为。
- WebRTC 白名单 endpoint、DataChannel 名称、RTP 细节。
- `response.cancel` 后是否仍可能收到旧 response 的音频/文本 delta。
- `response.audio_transcript.done`、`response.audio.done`、`response.done` 在正常、打断、错误路径下的相对顺序。

这些未知项不能阻止我们做静态适配矩阵，但会影响最终兼容性测试。后续如果进入实现，这些会成为 live smoke / integration test 的对象。

## 7. 我认为报告还存在的边界

这份报告已经足够作为下一轮实现输入，但它不是实现计划，也没有声称完成实现设计。它的 `后续实现输入` 是对象级要求，不是步骤顺序。这符合你的要求。

报告中有一个需要注意的点：subagent 提到“本地 `.venv` 路径在本次文件枚举中未返回文件；因此报告中的 Vision Agents 代码结论以上游 clone 为直接源码依据”。这不影响对 `external/Vision-Agents` 的源码审计，但如果后续实施，仍需要在进入代码修改前确认 TideSync 当前安装包和 upstream clone 的具体差异。这个确认属于实现前的版本基线动作，不是本轮调查缺口。

另一个边界是 Function Calling。报告根据本地 API 文档确认 Qwen3.5 Realtime 支持 Function Calling，但本地文档没有展开工具事件 schema。因此报告只能把“Vision Agents adapter 没有工具能力面”判为缺失，把“具体 tool event 字段”归为 live/API 文档未知。这个归档是正确的：不能编造事件字段。
