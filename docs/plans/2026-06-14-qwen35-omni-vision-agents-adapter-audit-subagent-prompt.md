## Required Method

### Step 1: Establish The Official Capability Contract

Extract an authoritative capability matrix from the Qwen3.5 Omni Realtime documentation. The matrix must distinguish:

- Model family and exact model codes.
- WebSocket connection contract.
- WebRTC connection contract.
- Session configuration fields.
- Input modalities and their transport rules.
- Output modalities and their transport rules.
- Server VAD and semantic VAD.
- Manual mode.
- User audio transcription events.
- Assistant text-only response events.
- Assistant audio response events.
- Response lifecycle events.
- Interruption and cancellation events.
- Image/video input rules.
- Search capability.
- Function Calling capability.
- Voice control capability.
- Voice and audio format constraints.
- Context, turn, duration, and session limits.
- Usage and billing fields exposed in response events.
- Error behavior and recoverability statements documented by the API.

Do not collapse WebSocket and WebRTC into one generic "realtime" column. They have different transport semantics.

- Default voice.

- Audio input format conversion.
- Audio output format interpretation.
- Video/image forwarding implementation.
- VAD mode support.
- Manual mode support.

- Local audio output signaling into Vision Agents core.
- Transcript event signaling into Vision Agents core.
- Speech started/ended event signaling into Vision Agents core.
- Response lifecycle signaling into Vision Agents core.
- Tool/search/function calling support, if any.
- Error and reconnect behavior.
- Metrics and observability behavior.
- Tests covering Qwen realtime behavior.

### Step 3: Inspect Vision Agents Core Realtime Semantics

Trace how the Qwen adapter's emitted events are consumed by Vision Agents core. Identify the core objects and event classes involved in:

- Agent speech started/ended.

## Hypotheses To Verify

These are hypotheses, not conclusions. Verify or reject each one with evidence:

1. Vision Agents Qwen realtime defaults may still use older Qwen/Qwen3-era assumptions, such as `gummy-realtime-v1` or non-Qwen3.5 defaults.
2. The adapter may support only `server_vad`, not `semantic_vad`.
3. The adapter may not expose manual mode even though WebSocket supports manual mode.
4. The adapter may send WebSocket `input_audio_buffer.append` and `input_image_buffer.append`, not Qwen native WebRTC RTP.
5. The adapter may not handle `input_audio_buffer.speech_stopped`.
6. The adapter may not emit Vision Agents `RealtimeUserSpeechStarted` and `RealtimeUserSpeechEnded`.
7. The adapter may not emit `RealtimeAgentSpeechStarted` and `RealtimeAgentSpeechEnded`.
8. The adapter may not emit `RealtimeAudioOutputDone` for `response.audio.done` or `response.done`.
9. The adapter may call `response.cancel` but fail to trigger local audio output clearing through Vision Agents core.
10. The adapter may not guard against stale `response.audio.delta` after interruption.
11. The adapter may not handle `response.audio_transcript.done`.
12. The adapter may ignore user transcription delta events.
13. The adapter may not expose search configuration or search usage fields.
14. The adapter may not expose Function Calling tool schema registration, tool-call events, tool execution, or tool-result return.
15. The adapter may not expose Qwen3.5 voice-control behavior as an observed state or configuration surface.
16. The adapter may not expose usage accounting from `response.done`.
17. The adapter's image/video timing state may be too weak to satisfy Qwen's `append image after audio` rule across turns and track reconnects.
18. The adapter may not distinguish WebSocket capability coverage from WebRTC capability coverage.
19. Vision Agents core may already contain event classes and audio flush mechanisms that the Qwen adapter does not use.
20. The old zipball may be useful only for the client-side playback interruption pattern, not for complete Qwen3.5 adapter design.
