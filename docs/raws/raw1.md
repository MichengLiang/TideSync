# 实时全模态视频通话 AI：开源技术方案调查与推荐

我已经按你给的目标做了一轮交叉调查：目标不是单纯“摄像头 + 麦克风 + 模型 API”，而是要接近 ChatGPT 高级语音、Gemini Live、豆包视频通话这种消费级体验：用户开摄像头和麦克风，AI 能实时听、实时看、用语音自然回应，支持打断、上下文连续、必要时接工具和联网信息。

下面是我的结论。先说最重要的判断：**目前没有一个完全开源、拿来就能一键复刻 ChatGPT / Gemini Live / 豆包视频通话的完整成品**。这些头部产品的模型、调度、端侧策略、音视频 QoS、记忆和安全策略大多是闭源系统。但现在开源生态已经足够拼出一个可用且工程上合理的版本，尤其你已经有了阿里云百炼的 `qwen3.5-omni-flash-realtime` 这个基座模型，难度大幅降低。真正要选的是：用哪一套实时音视频与 Agent 框架承接前后端媒体流、会话编排、客户端、监控、扩展能力。

我的推荐排序是：

1. **首选：LiveKit + 自定义 Qwen Realtime Adapter**。这是最接近生产级的路线，适合做 Web / 移动端“AI 视频通话”产品。
2. **第二选择：Pipecat + 自定义 Qwen Realtime Service**。适合快速拼多模型流水线、验证 Agent 行为、做服务端音视频/语音管线实验。
3. **强视觉场景可考虑：Stream Vision Agents**。适合健身私教、动作纠正、商品识别、现场导览、低延迟视觉处理这类“视频理解是核心”的场景。
4. **如果你未来想和 Agora 深度绑定：TEN Framework**。它定位也很准，但我会把它放在候选而不是第一落地方案。
5. **不建议从 mediasoup / Janus / Pion / aiortc 这些底层 WebRTC 组件直接重造产品**，除非你的目标是研究底层实时通信系统，而不是尽快做出 AI 视频通话体验。

## 一、先校准产品目标：闭源产品到底做了什么

### 1. ChatGPT 高级语音 + 视频

OpenAI 官方材料能确认两件事。第一，GPT-4o 这一代的方向就是原生多模态：文本、音频、图像、视频可以作为输入，输出可以是文本、音频、图像；OpenAI 公布过音频输入响应最快 232ms、平均 320ms 这一组指标。第二，ChatGPT Voice FAQ 明确提到移动端语音对话中可以按摄像头按钮共享设备视频，Release Notes 也记录了 advanced voice 中 real-time video、screen share、image upload 的 rollout。OpenAI Realtime API 官方文档也给了 WebRTC 接入方式，这说明 OpenAI 在开发者侧也把 WebRTC 当成浏览器实时语音 Agent 的标准入口。

需要注意：ChatGPT 产品本身闭源，官方公开 API 与 ChatGPT App 的内部产品栈不是一回事。我们能学到的是范式：**端侧实时媒体输入 + 原生实时模型 + WebRTC/低延迟音频 + 打断 + 上下文管理**，但无法直接复用 ChatGPT 的产品实现。

参考：
- https://openai.com/index/hello-gpt-4o/
- https://help.openai.com/en/articles/8400625-voice-mode-faq
- https://developers.openai.com/api/docs/guides/realtime-webrtc

### 2. Google Gemini Live / Project Astra

Google 官方资料同样能确认产品方向。Gemini Live 已经支持 camera 和 screen sharing；Google AI for Developers 的 Gemini Live API 文档写得更直接：Live API 面向低延迟、实时语音和视觉交互，处理连续音频、图像、文本流，输出即时的类人语音响应。Google Cloud 的 Agent Platform 文档还用“audio, video, or text”描述 Live API 的输入。

这对我们有两个启发：第一，消费级体验不是“拍一张图问答”，而是持续视觉上下文。第二，Google 自己也把这类能力产品化为 Live API，而不是要求开发者自己拆成 STT + VLM + TTS。

参考：
- https://blog.google/products-and-platforms/products/gemini/gemini-live-android-tips/
- https://ai.google.dev/gemini-api/docs/live-api
- https://deepmind.google/models/project-astra/

### 3. 豆包视频通话

豆包这边，公开信息比 OpenAI/Google 分散。字节 Seed 官方能确认的是豆包实时语音大模型：端到端语音对话、低时延、可打断、情绪承接。视频通话能力主要靠第三方实测和媒体报道交叉验证，例如每日经济新闻/证券时报转载报道提到豆包 App 上线视频通话，基于视觉推理模型并支持联网搜索，记者实测包括水果成熟度识别、持续记忆、逻辑推理等。

这里我的判断是：豆包视频通话产品存在，且确实进入了“用户摄像头画面 + 语音对话 + 视觉推理/搜索”的形态。但“0.5 秒延迟”“端侧如何抽帧调码率”等具体工程细节，公开可信来源不如 OpenAI/Google 官方文档完整，应该当作产品体验参考，而不是当作可精确复刻的技术规格。

## 二、你手上的 Qwen Realtime 基座模型，已经覆盖了最难的一层

阿里云官方文档明确写到：Qwen-Omni-Realtime 是实时音视频聊天模型，能理解流式音频与图像输入，例如从视频流实时抽取连续图像帧，并实时输出文本与音频。它支持 WebSocket 和 WebRTC 两种接入：WebSocket 适合服务端集成和快速接入；WebRTC 适合浏览器低延迟语音场景，音频走 UDP，内置回声消除和降噪。

更关键的是视频输入方式：

- WebSocket：客户端发送 `input_audio_buffer.append` 和 `input_image_buffer.append`，也就是音频流 + 图像帧。图像可以来自本地文件，也可以从视频流实时采集。
- WebRTC：建连时添加音频轨道和视频轨道，音频和视频通过 RTP 媒体通道传到服务端。官方说明 WebRTC 视频通过视频轨道发送画面帧，不需要 `input_image_buffer.append`。
- 官方 FAQ 还建议视频通话场景可以抽帧，频率建议 1 张/秒。这一点非常重要，因为它揭示了大多数“实时视觉对话”的真实工程做法：并不是每秒 30 帧全部进模型，而是实时通话媒体保持流畅，模型侧按策略采样关键帧/低频帧。

Qwen3.5-Omni-Realtime 还有几个对产品很重要的能力：联网搜索、Function Calling、语义打断、语音控制、服务端 VAD、长会话上限。阿里云文档显示会话最长 120 分钟；`qwen3.5-omni-flash-realtime` 的视频上下文最大轮次和最大时长也有明确限制，视频帧不是无限记忆，要做产品记忆仍需额外设计。

这意味着你不需要自己拼 STT + VLM + LLM + TTS 的完整链路来证明可行性。你现在有的是“端到端实时全模态模型 API”。剩下最重要的工程任务是：**如何把浏览器/移动端的摄像头麦克风、WebRTC 房间、后端 Agent、Qwen Realtime 会话、工具调用、前端体验稳定地连接起来**。

参考：
- https://help.aliyun.com/zh/model-studio/realtime
- https://help.aliyun.com/zh/model-studio/omni-realtime-python-sdk
- https://help.aliyun.com/zh/model-studio/server-events

## 三、生态里的开源方案分层

不要把“开源解决方案”理解成一个项目。这个系统至少有五层：

1. **客户端层**：Web / iOS / Android，负责摄像头、麦克风、扬声器、权限、屏幕共享、通话 UI、字幕、打断提示。
2. **实时媒体传输层**：WebRTC、SFU、TURN、弱网处理、音视频轨道、房间、设备切换。
3. **Agent 编排层**：管理用户会话、模型会话、上下文、工具调用、事件、VAD、打断、观测日志。
4. **模型接入层**：Qwen Realtime WebSocket/WebRTC，或者 OpenAI Realtime、Gemini Live、级联 STT-LLM-TTS。
5. **视觉增强层**：抽帧策略、YOLO/Roboflow/OpenCV/姿态估计、物体跟踪、场景状态缓存。

闭源产品把这五层打包成一个 App。开源世界里，成熟项目通常只覆盖其中两三层。真正合理的方案是选择一个强框架作为主干，然后补齐 Qwen Adapter。

## 四、候选方案评估

### 方案 A：LiveKit + 自定义 Qwen Realtime Adapter（首选）

LiveKit 是我最推荐的主干。它不是单纯的 AI 框架，而是一整套实时音视频基础设施：开源 WebRTC SFU、房间、token、客户端 SDK、服务端 SDK、SIP、录制、Ingress/Egress、Agent SDK。LiveKit 官方对 Agents 的定位是“realtime framework for voice, video, and physical AI agents”，支持把 Python/Node 程序作为实时参与者加入 LiveKit 房间，并通过 WebRTC 与前端通信。官方文档还明确说 LiveKit Agents 支持 voice、video、text，可处理实时输入并输出任意模态，支持 turn detection、interruptions、LLM orchestration、tools、多 Agent handoff、Kubernetes 部署。

我查了 GitHub 当前数据：`livekit/livekit` 约 19.2k stars，`livekit/agents` 约 10.9k stars，两个仓库截至 2026-06-12 仍有近期提交。许可证是 Apache-2.0。这说明它不是小众玩具，而是已经足够成熟的实时通信底座。

LiveKit 对你的价值：

- 前端 Web / iOS / Android / Flutter / React Native 生态完整。
- 摄像头、麦克风、屏幕共享、房间、弱网和音视频轨道不用你从零写。
- Agent 可以作为房间里的一个参与者，天然符合“用户与 AI 视频通话”的产品模型。
- 未来如果要多人房间、人工介入、客服、远程协助、录制、审计，LiveKit 的架构能承接。
- 你可以先用 LiveKit Cloud 快速跑通，也可以自托管 LiveKit Server。

短板也明确：LiveKit 官方目前没有现成的 `qwen3.5-omni-flash-realtime` Realtime 插件。LiveKit 有 OpenAI Realtime、Gemini 等集成，但 Qwen Realtime 需要自己写 adapter。这个 adapter 的工作包括：从 LiveKit 房间接收用户音频/视频轨道，按 Qwen 要求转成 WebSocket 音频流和图像帧，或直接研究 Qwen WebRTC 接入；把 Qwen 返回的音频流发布回房间；把字幕、转录、工具调用事件写回 data channel 或后端事件流。

我认为这是合理工作量，不是“造轮子”。因为 LiveKit 已经把最难的 WebRTC 产品基础设施做了，Qwen Adapter 只是模型接入层。

适合场景：正式产品、Web/移动端、视频通话 UI、后续多用户/客服/远程协助/录制/部署扩展。

参考：
- https://docs.livekit.io/agents/
- https://docs.livekit.io/agents/multimodality/vision/
- https://github.com/livekit/livekit

### 方案 B：Pipecat + 自定义 Qwen Realtime Service（第二推荐）

Pipecat 是 Daily 发起的开源实时语音/多模态 Agent 框架。官方定位很清楚：构建 voice and multimodal AI agents，可以 see、hear、speak。它的优点是 pipeline 抽象强，能把 transport、VAD、STT、LLM、TTS、frame processor、context aggregation 等环节串成低延迟流水线。Pipecat 文档里已经有 Gemini Live 服务，支持音频、视频、文本，多模态处理、实时流、VAD、Function Calling、上下文管理。它也有 Qwen LLM Service，但目前这个 Qwen 集成主要是 OpenAI-compatible 文本 LLM，不等于 Qwen Omni Realtime 音视频 API。

GitHub 当前数据：`pipecat-ai/pipecat` 约 12.8k stars，BSD-2-Clause，近期仍活跃。这说明 Pipecat 的社区活跃度和工程质量都不错。

Pipecat 对你的价值：

- 非常适合做“模型/管线实验台”：可以替换 OpenAI、Gemini、Qwen、Deepgram、Cartesia、ElevenLabs 等组件。
- 如果你未来要比较“Qwen 端到端”与“STT + LLM + TTS 级联”的延迟和成本，Pipecat 很方便。
- 它的 frame/pipeline 模型适合把图像帧、音频帧、转录、模型事件作为统一流处理。
- 服务端 Python 生态友好，和你当前 workbench 的 Python 主环境也匹配。

短板：它不像 LiveKit 那样本身就是完整 SFU/房间产品底座。Pipecat 可以接各种 transport，但如果目标是消费级视频通话 App，仍需要你选择 RTC provider 或前端连接方式。另一个短板同样是 Qwen Realtime 需要自定义 service，现成 Qwen LLM 不能直接覆盖音视频实时通话。

适合场景：原型验证、模型管线比较、快速做 Agent 行为实验、服务端多模型编排、未来可能混合 STT/LLM/TTS。

参考：
- https://docs.pipecat.ai/overview/introduction
- https://docs.pipecat.ai/api-reference/server/services/s2s/gemini-live
- https://docs.pipecat.ai/api-reference/server/services/llm/qwen

### 方案 C：Stream Vision Agents（强视觉场景候选）

Vision Agents 是 Stream 团队新推的开源 Python 框架，定位非常贴合“实时视频 AI Agent”。官方说它是 open-source Python framework for voice and video AI agents，支持实时 WebRTC、25+ integrations、native tool calling、pluggable vision pipelines。它的示例代码直接就是 Agent + Gemini Realtime + YOLOPoseProcessor 这种模式。Stream 官方博客还说 Vision Agents 支持 OpenAI Realtime 和 Gemini Live，并能处理 tracks、video subscriptions、response type 转换等复杂性。

GitHub 当前数据：`GetStream/Vision-Agents` 约 7.9k stars，Apache-2.0，近期活跃。它比 LiveKit/Pipecat 年轻，但增长快，而且主题非常贴合“视频 Agent”。

Vision Agents 对你的价值：

- 它不是泛泛的 voice agent，而是偏“实时视觉 AI 应用”。
- 内置思路就是把 YOLO、Roboflow、pose detection、Gemini/OpenAI Realtime 等组合起来。
- 如果你做健身私教、动作纠正、球类教练、商品识别、现场导览、监控告警，它的处理器机制更贴近业务。
- 它能避免你把所有视觉任务都丢给 Qwen。比如姿态检测、物体框、计数、OCR/检测可以本地或边缘先做，Qwen 负责语言理解、推理和语音回应。

短板：它强绑定 Stream 的 realtime transport 体验，虽然官方说也能与其他 WebRTC 基础设施配合，但最顺的路线显然是 Stream 账号和 Stream edge。另一个问题是它更年轻，生态稳定性和生产部署经验还需要谨慎评估。接 Qwen Realtime 也同样需要自定义 integration。

适合场景：视频理解是核心卖点，而不只是“AI 偶尔看一眼摄像头”。例如健身教练、桌面操作指导、维修指导、商品验真、体育评论、安防辅助。

参考：
- https://visionagents.ai/
- https://getstream.io/vision-agents/
- https://github.com/GetStream/Vision-Agents
- https://getstream.io/blog/vision-agents-by-stream/

### 方案 D：TEN Framework（Agora 生态候选）

TEN 是另一个值得关注的实时多模态 conversational AI 框架。官方文档写明它是 open-source framework for real-time multimodal conversational AI，支持模块化、扩展、低延迟、多语言扩展。TEN Agent Examples 明确提到 Gemini 2.0 Live 和 OpenAI Realtime API，架构上编排 ASR、LLM、TTS、RTC 组件，支持 full-duplex conversation 和自然打断。Agora 页面也把 TEN 描述为可创建能 see、hear、speak 的实时多模态 AI，并且由 Agora 和社区支持。

GitHub 当前数据：`TEN-framework/ten-framework` 约 10.7k stars，近期活跃。但我用 GitHub API 查到 license 字段是 `NOASSERTION`，正式商用前需要人工核对仓库许可证和各组件许可证。

TEN 的价值：

- 它强调模块化扩展，适合做复杂 Agent 运行时。
- 与 Agora 生态有天然关联，Agora 在实时音视频商业化和全球网络上经验很强。
- 它自带 VAD、Turn Detection 等组件方向，适合 full-duplex 语音体验。

短板：从公开资料看，TEN 的文档和生态成熟度不如 LiveKit/Pipecat 清晰。它适合进入 PoC 候选，但我不建议把第一版产品押在 TEN 上，除非你已经决定使用 Agora 作为 RTC 基础设施，或者你明确喜欢它的 extension 架构。

参考：
- https://theten.ai/docs
- https://theten.ai/docs/ten_agent_examples/overview
- https://prod.agora.io/en/open-source/ten-framework

### 方案 E：直接用 Qwen 官方 WebRTC/WebSocket 示例 + 自己写前端后端

这是最快验证模型的办法。阿里云官方文档本身已经有 Python SDK、WebSocket、WebRTC、函数计算模板。你可以先不引入 LiveKit/Pipecat，直接做一个最小 Web App：浏览器拿摄像头麦克风，服务端连接 Qwen Realtime，音频流和抽帧图像发给 Qwen，Qwen 音频返回给浏览器播放。

优点：快、依赖少、最能验证 Qwen 模型体验、成本最低。缺点：一旦要产品化，你会很快遇到房间管理、弱网、设备切换、回声消除、断线恢复、监控、权限、安全、TURN、移动端兼容等问题。也就是说，这条路线适合“第一周验证”，不适合“长期产品主干”。

我建议把它作为第 0 阶段，而不是最终架构。

## 五、底层 WebRTC 轮子：为什么不建议从它们开始

你会看到很多底层项目：mediasoup、Janus、Pion、aiortc。

- **mediasoup**：强大的 Node/C++ WebRTC SFU，适合团队有 RTC 专家、想高度自定义房间和媒体路由。
- **Janus Gateway**：老牌 C WebRTC server，插件体系成熟，但产品层开发体验不如 LiveKit 现代。
- **Pion WebRTC**：Go 语言 WebRTC 实现，LiveKit server 也基于 Pion。适合写底层媒体服务，不适合直接当应用框架。
- **aiortc**：Python WebRTC/ORTC 实现，适合研究、单机 demo、服务端点对点 WebRTC。阿里云文档的 WebRTC Python 示例也能看到类似思路。但它不提供完整产品级 SFU/房间体系。

这些都不是坏东西，但它们解决的是底层通信问题，不是 AI 视频通话产品问题。你的目标是搭出 ChatGPT / Gemini Live / 豆包式体验，所以不应该一开始就掉到底层媒体协议里。除非你有明确的定制 RTC 需求，否则选 LiveKit 这类完整实时栈会更稳。

参考：
- https://github.com/livekit/livekit
- https://github.com/pion/webrtc
- https://github.com/aiortc/aiortc

## 六、关于“全开源本地模型”的现实判断

你现在用的是阿里云百炼 Qwen3.5-Omni-Flash-Realtime，是云 API。问题是：能不能完全开源、本地自托管出同样体验？

Qwen3-Omni 开源仓库确实存在，官方说明它是端到端 multilingual omni-modal foundation model，能处理文本、图像、音频、视频，并输出实时流式文本和自然语音。vLLM-Omni 也已经有 Qwen3-Omni serving 文档。但是从第三方技术教程的延迟评测看，现实还没那么理想：DashScope Realtime API 可以做到约 702ms 平均 TTFA；vLLM Thinker 自托管可以很快做 audio-to-text/text generation，但没有音频输出；Transformers 全量 Thinker+Talker 自托管音频输出非常慢，不适合实时。该教程的结论是：优化过的 Talker 实时音频生成目前主要在 DashScope cloud API 中可用，完全自托管的端到端实时语音还不成熟。

所以我建议：**第一版不要追求全本地开源模型**。用 DashScope Realtime 做主模型，把系统架构设计成 provider-adapter 模式，将来如果 vLLM-Omni 的 Talker serving 真正成熟，再替换模型层。这样不会把产品进度卡死在本地推理性能上。

- https://docs.vllm.ai/projects/vllm-omni/en/latest/user_guide/examples/online_serving/qwen3_omni/

## 七、我推荐的落地架构

### 第一阶段：最小可用原型

目标：验证 Qwen3.5-Omni-Flash-Realtime 的真实交互体验，而不是先搭大系统。

做法：

- Web 前端打开摄像头和麦克风。
- 音频以 16k PCM 或 SDK 要求格式送 Qwen。
- 视频侧按 1 fps 抽帧，先走 WebSocket `input_image_buffer.append`。
- Qwen 输出音频流，前端播放；同时显示转录和回复文本。
- 打断先用 Qwen 服务端 VAD/semantic VAD 能力，前端检测到用户说话时停止当前播放。
- 记录每轮 first text delay、first audio delay、端到端响应时间、丢帧/断线情况。

这个阶段可以直接参考阿里云官方 SDK 示例，不必先引入 LiveKit。验收标准是：在真实网络下，是否能做到“可连续对话、可看画面、能打断、延迟可接受”。

### 第二阶段：LiveKit 产品化骨架

目标：把原型升级为“像通话产品”的结构。

建议架构：

- Browser / Mobile App：LiveKit Client SDK 发布 mic/camera/screen tracks。
- LiveKit Server / Cloud：负责房间、音视频轨道、弱网、TURN、token、参与者管理。
- Qwen Agent Service：作为 LiveKit room participant 加入房间。
- Adapter：订阅用户音频轨道和视频轨道；音频转 Qwen 输入；视频按策略抽帧或尝试 Qwen WebRTC 视频轨道；接收 Qwen 音频输出后发布为 agent audio track；文本/字幕/状态通过 data channel 发给前端。
- Tool Service：天气、搜索、业务 API、RAG、地理位置、商品库等工具。
- Observability：记录会话 id、延迟、首包时间、VAD 事件、模型错误、工具调用、用户反馈。

这就是我认为最合理的主线。

### 第三阶段：视觉增强

Qwen 可以理解画面，但产品不能把所有视觉任务都丢给大模型。要想接近豆包/Gemini Live 的“稳定可用”，需要加视觉状态层：

- 抽帧策略：默认 1 fps；用户问“看这里/这个是什么”时临时提高到 2-5 fps；静止画面降频。
- 关键帧缓存：保存最近 N 秒的帧摘要，而不是保存所有图像。
- 视觉状态表：桌面上有哪些物体、用户正在指什么、上一次提到的物体是什么。
- 轻量 CV：YOLO/Roboflow/OpenCV/姿态估计先做结构化检测，再让 Qwen 解释。
- 场景触发：健身动作不标准、物体进入画面、屏幕出现错误码等，可以由本地视觉处理器触发模型回应。

如果你的产品偏视觉教练/动作纠正，这一层可以考虑引入 Vision Agents。如果只是普通 AI 视频通话，LiveKit + 自己的 frame sampler 足够。

## 九、最终建议

如果你问“现在最主流、成熟、可落地的开源技术方案是什么”，我的答案是：

**LiveKit 做实时音视频与产品骨架，Qwen3.5-Omni-Flash-Realtime 做实时全模态模型，自写一层 Qwen Realtime Adapter；视觉场景重时再引入 Vision Agents 或自研轻量视觉处理器；Pipecat 保留为模型管线实验和备选 Agent 编排框架。**

这条路线的优点是稳：LiveKit 解决通话产品最麻烦的 WebRTC 和多端 SDK，Qwen 解决最难的端到端多模态理解和语音输出，你只在中间写必要胶水层和业务逻辑。它不会被某个年轻框架绑死，也不会让你从 mediasoup/Pion 这种底层通信开始重造整个产品。

我不建议的路线是：

- 不建议第一版追求全本地 Qwen3-Omni 自托管实时语音输出。现在云端 DashScope Realtime 明显更现实。
- 不建议第一版直接基于 mediasoup/Janus/Pion 从零写产品层，除非你团队本身就是 RTC 团队。
- 不建议把普通 Qwen OpenAI-compatible Chat Completions 当成 Realtime Omni API 使用，它只能覆盖文本 LLM，不等于实时音视频模型。
- 不建议只做“每隔几秒截图 + 普通 VLM + TTS”的假实时方案，体验上会明显输给 ChatGPT/Gemini/豆包式通话。

工程上最关键的三个风险是：

1. **Qwen Adapter 的实时音频格式和打断处理**：必须认真处理音频采样率、播放缓冲、用户说话时清空输出、服务端 VAD 事件。
2. **视频抽帧和上下文成本**：视觉不是帧率越高越好，过高会烧 token、增加延迟、污染上下文。默认 1 fps 是合理起点。
3. **产品侧状态管理**：用户需要知道 AI 是否正在听、正在看、正在回答、是否被打断；否则技术可用但体验会混乱。

如果下一步进入实现，我建议先做一个 `experiments/qwen-realtime-video-call/` 小项目：第一个里程碑只接 DashScope WebSocket + 浏览器抽帧；第二个里程碑再接 LiveKit。这样路径最短，也最容易用真实体验决定后面要不要上完整框架。

5. 架构适配真值：由接口匹配关系承载。例：某框架是否允许自定义模型 service/adapter、是否能接收视频轨道或抽帧。
6. 项目环境真值：由你的账号、网络、甲方需求、成本、端设备、部署约束承载。这个才需要在你的环境中测。

我之前把“官方能力真值”和“项目环境真值”混成一类，于是才会提出不必要的验证动作。正确做法是：先通过资料确认前五类真值，只有当第六类成为决策前提时才运行测试。否则就是你说的“非得看 CPU 执行机器码才安心”。

8. **TEN Framework 值得跟踪，但在当前证据下不是第一推荐。** 它定位是实时多模态 conversational AI，和 Agora 生态关联强，但许可证字段、Qwen 适配、文档成熟度都还需要更谨慎地审查。

9. **OpenAvatarChat 等数字人项目可作为相关参考，但不是当前对象。** 它们证明社区已经在把 Qwen Omni 放进实时语音/数字人交互系统，但对象是“数字人对话”，不是“用户摄像头现场给 AI 看”的视频通话。

10. **真正的缺口不是“有没有理论指导”，而是几个明确工程缺口：Qwen Realtime 与所选 RTC/Agent 框架的契约整合、用户打断与音频缓冲一致性、视觉帧策略与上下文管理、前端公共投影、成本/隐私/观测治理。** 这些不是爱迪生式发明灯泡，但也不是可以忽略的胶水。

## 一、当前人工制品的正面定义

当前对象可以定义为：**消费级实时全模态 AI 视频语音通话系统**。

它不是普通聊天机器人，不是语音识别工具，不是拍照问答，不是会议软件，也不是数字人本身。它的对象成立条件来自用户侧体验：用户打开摄像头和麦克风，把现实现场持续暴露给 AI；AI 同时接收语音和视觉现场，在同一个会话上下文中理解用户问题、画面状态、指代关系和动态变化，并用自然语音作答。

如果用 Zave 与 Jackson 的 `Domain ∧ Specification ⊨ Requirements` 表达：

- `Domain`：用户有手机或浏览器设备；设备有摄像头、麦克风、扬声器；用户处于现实场景中；网络存在延迟和抖动；用户希望像视频通话一样自然询问；模型服务有协议、限流、计费和上下文限制；前端必须处理权限、播放、打断、断线和隐私。
- `Requirements`：用户能自然地让 AI “看见我正在看的东西、听见我说的话，并及时用语音回应”；对话可连续、可打断、可基于画面追问；系统行为对用户可理解。
- `Specification`：我们可控的部分，包括前端通话界面、RTC/传输层、Agent 服务、Qwen Realtime 接入、视频帧策略、工具调用、状态投影、错误处理、日志和成本策略。

这里的构成性条件不是“用了哪个框架”，而是下面这些能力：

1. **设备输入成立**：用户能授权摄像头和麦克风，系统能持续采集音频和视觉现场。
2. **实时媒体成立**：音频不能只是录完上传，视频不能只是偶尔手动拍照；系统要支持持续流式输入或按明确策略抽帧。
3. **同一会话上下文成立**：语音、视觉、工具结果、历史对话不能彼此割裂。用户说“这个”“刚才那个”“你看我手里这个”时，系统需要有指代承接的可能。
4. **自然语音输出成立**：AI 的主输出是语音，文本字幕可以作为辅助投影，不是唯一回应。
5. **可打断性成立**：用户重新说话时，系统应停止或取消当前回答，避免“机器人必须播完”的僵硬体验。
6. **视觉理解成立**：系统能识别画面中的物体、文字、动作、空间关系或用户操作；对于动态场景至少要有持续抽帧或视频轨道输入机制。
7. **公共投影成立**：用户要能看见系统状态：正在听、正在看、正在思考/回答、被打断、无权限、断线、网络差、摄像头关闭。
8. **失败语义成立**：权限失败、模型限流、网络断开、音频设备占用、视频帧过大、API 错误，都要有可解释行为。
9. **成本边界成立**：音频、视频帧、输出语音、工具调用都会计费；系统必须能控制抽帧频率、会话时长、静音输入和工具调用频率。
10. **隐私边界成立**：摄像头和麦克风是高敏输入；用户必须知道何时被采集、何时停止、数据是否保存、是否上传云模型。

这些条件缺失以后，系统就会从“消费级实时全模态视频通话 AI”降级为其他对象。例如只有文字输出，就是多模态问答；只有拍照上传，就是图片问答；只有语音无视频，就是语音助手；只有 WebRTC 房间无模型，就是视频通话工具；只有模型 API 无前端状态，就是后端能力样例。

## 三、Qwen3.5-Omni-Flash-Realtime 的正确位置

你已经找到了这项任务里最关键的模型基座。我的查证把它放在以下位置：**它是多模态模型会话职责的核心手段，不是完整产品本身。**

阿里云百炼 Qwen-Omni Realtime 官方文档确认了几个关键能力：

- 它是实时音视频聊天模型。
- 支持 WebSocket 和 WebRTC 接入。
- WebSocket 适合服务端集成和快速接入。
- WebRTC 适合浏览器端、低延迟语音场景，音频通过 UDP 传输，并利用浏览器/RTC 栈的回声消除和降噪能力。
- 支持实时音频输入、视觉输入、文本/音频输出、服务端 VAD、语音转录、语音输出等事件式交互。
- 通过事件协议可以发送 `input_audio_buffer.append`、`input_image_buffer.append`，接收 `response.audio.delta`、转录完成、音频转录增量等事件。

你提供的模型详情还补充了业务上非常有价值的信息：`qwen3.5-omni-flash-realtime` 支持文本、图片、音频、音视频理解与交互，支持 60+ 语言音频输入、30+ 语言语音输出、WebSearch、复杂 FunctionCall、智能语义打断，具备 256K 上下文、RPM/TPM 限流和清晰计费。

这里的推导是：

- 如果没有 Qwen Realtime，我们需要自己拼 STT、VLM、LLM、TTS、VAD、打断、音频缓冲、上下文聚合，系统复杂度会大幅上升。
- 有 Qwen Realtime 后，模型层可以被视作一个强外部人工制品：它直接提供实时多模态会话能力。
- 但 Qwen Realtime 不负责完整消费级产品体验。它不替你完成前端通话 UI、多端 SDK、RTC 房间、用户状态投影、隐私提示、业务工具授权、日志治理、成本策略。

所以，Qwen 在整体结构中是“模型会话引擎”，不是“完整视频通话产品”。

- https://help.aliyun.com/zh/model-studio/omni-realtime-python-sdk
- https://help.aliyun.com/zh/model-studio/server-events

## 四、是否存在完整开源成品

这里要把命题说清楚。

如果“完整开源成品”指：一个开源项目自己提供前端摄像头麦克风、WebRTC/RTC、实时视频 Agent、端到端多模态模型、语音输出、打断、上下文、工具调用、部署治理，并且不依赖闭源云模型或商业 RTC 网络，那么目前没有查到。

如果“开源技术解决方案”指：开源框架 + 云端实时多模态模型 + 可用 RTC/edge 服务 + 少量业务适配，那么已经存在非常接近的方案。其中最接近你当前 Qwen 基座的是 Vision Agents + Qwen Realtime 插件。

因此结论不能写成简单的“有”或“没有”。精确结论是：

- **全开源、全本地、完整闭环成品：未查到。**
- **开源框架承载 + 云端 Qwen Realtime 模型 + 商业/自建 RTC 传输：可行，并且已有接近成品的轮子。**
- **OpenAI/Gemini/豆包同等级闭源产品体验：可作为目标坐标，但不能直接复用其内部实现。**

## 五、开源候选方案查证

### 1. Vision Agents + Qwen 插件：当前最贴近 Qwen 视频通话目标的现成入口

这是本轮调查最重要的新发现。

Vision Agents 是 Stream 团队推出的开源 Python 框架，定位是 voice and video AI agents。GitHub 仓库 `GetStream/Vision-Agents` 当前约 7.9k stars，Apache-2.0，近期活跃。PyPI 上 `vision-agents` 当前版本为 `0.6.4`，摘要是 “Open video agents. Build low latency video and voice agents on any realtime edge network.”

更关键的是它的 Qwen 插件：

- 仓库路径：`plugins/qwen/`
- PyPI 包：`vision-agents-plugins-qwen`
- 当前版本：`0.6.4`
- 上传时间：2026-06-11
- 包摘要：`Qwen Omni plugin for vision agents`
- 依赖：`vision-agents`、`numpy`、`websockets>=15.0.1`
- 插件 README 明确写出能力：native audio output、built-in STT、server-side VAD、optional video frame support、WebSocket bidirectional realtime、interruption handling。

我直接查了插件代码。它不是文档口号，而是已经实现了核心事件闭环：

- `Qwen3RealtimeClient` 建立到 DashScope Realtime WebSocket 的连接。
- `send_audio` 将 RTC 中的 PCM 音频重采样为 16k mono PCM，并发送 `input_audio_buffer.append`。
- `send_frame` 将 JPEG 图像 base64 后发送 `input_image_buffer.append`。
- `Qwen3Realtime` 继承 Vision Agents 的 `Realtime` 抽象。
- `watch_video_track` 订阅视频轨道，通过 `VideoForwarder` 按 fps 取帧。
- `_send_video_frame` 将视频帧转 JPEG，默认可设 1280x720，并发送给 Qwen。
- `_process_events` 处理 `response.audio.delta`，把 Qwen 返回的音频转成 `PcmData` 发回 Agent 音频输出事件。
- 处理 `conversation.item.input_audio_transcription.completed`，发出用户语音转录。
- 处理 `response.audio_transcript.delta`，发出 AI 语音字幕增量。
- 处理 `input_audio_buffer.speech_started`，如果当前模型正在响应，就调用 `response.cancel` 取消当前回答。

这正好覆盖我们目标对象中的几项关键构成性条件：听、看、说、转录、打断、视频帧输入。

它的示例代码也非常接近目标形态：创建 `Agent`，使用 `getstream.Edge()`，LLM 使用 `qwen.Realtime(fps=1)`，加入 call 后打开 demo。也就是说，它已经把 Qwen Realtime 放进了一个实时视频 Agent 框架，而不只是裸 WebSocket client。

但边界也要说清楚：

- Vision Agents 依赖 Stream 的 edge/video 体系，使用它最顺的路径需要 Stream API credentials。
- 它是较新的框架，虽然活跃增长快，但生产案例和生态沉淀还不如 LiveKit。
- 插件默认 base URL 是 `dashscope-intl.aliyuncs.com`，而你当前控制台地域是中国内地，实际落地时需要核对中国内地 endpoint、模型名、音色、可用区和计费。
- 插件 README 默认模型名写过 `qwen3-omni-flash-realtime`，代码默认值是 `qwen3.5-omni-plus-realtime`，而你手上模型 code 是 `qwen3.5-omni-flash-realtime`。这不是理论障碍，但说明落地时必须显式配置模型名，不能盲用默认值。
- 插件说明 Qwen Realtime 不支持 text input，用户需要通过语音进入会话。这与目标对象一致，但如果产品还想支持文字输入，需要另建路径或另接模型。

所以我的判断是：**如果目标是用你当前的 Qwen Realtime 基座尽快搭出视频语音 AI 通话，Vision Agents + Qwen 插件是目前查到的最直接开源入口。**

来源：
- https://github.com/GetStream/Vision-Agents
- https://visionagents.ai/
- https://visionagents.ai/integrations/realtime/qwen
- https://pypi.org/project/vision-agents-plugins-qwen/

### 2. LiveKit + 自定义 Qwen Realtime Adapter：长期产品底座强候选

LiveKit 是更成熟的实时音视频基础设施。GitHub `livekit/livekit` 当前约 19.2k stars，Apache-2.0，Go；`livekit/agents` 当前约 10.9k stars，Apache-2.0，Python，近期活跃。PyPI `livekit-agents` 当前版本为 `1.6.0`，发布活跃。

LiveKit 的职责覆盖很强：

- WebRTC SFU 和房间。
- 多端 SDK：Web、iOS、Android、React Native、Flutter 等。
- token、participant、track、data channel、录制、SIP、Ingress/Egress。
- LiveKit Agents 让 AI Agent 作为房间参与者加入实时会话。
- 官方文档定位就是 realtime voice/video AI agents。

它对目标人工制品的意义是：如果要做真正产品，不只是 demo，用户端通话、设备切换、弱网、房间管理、录制、人工接管、多用户、移动端 SDK，都属于 LiveKit 的强项。

但我做了代码搜索：在 `livekit/agents` 仓库里没有查到 DashScope/Qwen Realtime 插件。也就是说，LiveKit 本身是非常好的实时通信和 Agent 底座，但要使用你手上的 Qwen Realtime，需要我们写 adapter。

这个 adapter 的职责不是科研，而是明确工程契约：

- 从 LiveKit 用户音频 track 中读取 PCM。
- 按 Qwen Realtime 要求转换采样率、通道和格式。
- 从用户视频 track 中按策略抽帧，转 JPEG/base64，走 Qwen `input_image_buffer.append`；或者研究 Qwen WebRTC 视频轨道直接接入，但这会更贴近协议层。
- 接收 Qwen `response.audio.delta`，发布为 LiveKit agent audio track。
- 把用户转录、AI 字幕、状态、错误通过 data channel 或后端事件投影给前端。
- 处理 `speech_started`、`response.cancel`、前端播放缓冲清理，形成可打断闭环。

因此 LiveKit 的判断是：**它不是当前 Qwen 接入的最短路径，但很可能是长期产品化最稳的 RTC/房间底座。**

来源：
- https://github.com/livekit/livekit
- https://github.com/livekit/agents
- https://docs.livekit.io/agents/
- https://pypi.org/project/livekit-agents/

### 3. Pipecat：优秀流水线框架，但当前不是 Qwen 视频通话最短路径

Pipecat 是 Daily 发起的开源 voice and multimodal conversational AI 框架。GitHub `pipecat-ai/pipecat` 当前约 12.8k stars，BSD-2-Clause，近期活跃。PyPI `pipecat-ai` 当前版本为 `1.3.0`。

它的强项是 pipeline 抽象：transport、VAD、STT、LLM、TTS、context、frame processor 可以组合。它适合比较不同模型服务、构建服务端实时语音流水线、在同一框架下替换 STT/LLM/TTS/Live API。

Pipecat 对 Gemini Live 这类 S2S 实时多模态服务有文档和服务支持。它也有 Qwen 服务，但我查到的 `src/pipecat/services/qwen/llm.py` 是普通 LLM service，不是 Qwen Omni Realtime 音视频服务。仓库内也没有查到 DashScope Omni Realtime 插件。

所以 Pipecat 的位置是：

- 如果要研究“端到端 Qwen Realtime”与“STT + LLM + TTS 级联”两条路径的工程差异，Pipecat 很合适。
- 如果要做复杂管线、多模型 A/B、替换供应商、加入专门的 CV processor，Pipecat 有价值。
- 如果要最快把 Qwen3.5-Omni-Flash-Realtime 接成视频通话，Vision Agents 当前更直接。

- https://github.com/pipecat-ai/pipecat

### 4. TEN Framework：值得观察，但当前证据不足以排第一

TEN Framework 定位为开源 conversational voice AI agents / realtime multimodal conversational AI framework，和 Agora 生态有明显关联。GitHub `TEN-framework/ten-framework` 当前约 10.7k stars，近期活跃。

它的价值在于：模块化、实时、多模态、Agora 生态、full-duplex conversation、OpenAI/Gemini 方向示例。

但当前不把它排第一，原因是：

- GitHub API 查到 license 字段为 `NOASSERTION`，商用前需要人工核对许可证和子模块许可证。
- 没查到成熟 Qwen/DashScope Realtime 插件证据。
- 文档和生态在当前任务里的确定性不如 Vision Agents/Qwen 插件和 LiveKit。

所以 TEN 是候选，不是本轮主推荐。

来源：
- https://github.com/TEN-framework/ten-framework
- https://theten.ai/docs

### 5. OpenAvatarChat：相关参考，不是当前主对象

OpenAvatarChat 是一个值得注意的相关项目。GitHub `HumanAIGC-Engineering/OpenAvatarChat` 当前约 3.5k stars，Apache-2.0，近期活跃。仓库里有 `src/handlers/llm/qwen_omni/llm_handler_qwen_omni.py`，使用 DashScope `OmniRealtimeConversation`，支持 Qwen Omni、音频、视频输入开关、转录、重连、心跳等。

这说明 Qwen Omni 已被社区用于实时交互系统。但 OpenAvatarChat 的对象更偏数字人/Avatar Chat：AI 以数字人形象与用户对话。它可以给我们参考 Qwen Omni handler、音频线程、重连、心跳、配置项，但不能直接等同于“用户打开摄像头把现场给 AI 看”的消费级视频通话产品。

来源：
- https://github.com/HumanAIGC-Engineering/OpenAvatarChat

### 6. mediasoup / Janus / Pion / aiortc：底层轮子，不是完整解

这些项目都很重要，但它们处在更底层：

- `mediasoup`：Node/C++ WebRTC SFU，约 7.2k stars，ISC。
- `Janus Gateway`：老牌 C WebRTC server，约 9.1k stars，GPL-3.0。
- `Pion WebRTC`：Go WebRTC 实现，约 16.5k stars，MIT。
- `aiortc`：Python WebRTC/ORTC，约 5k stars，BSD-3-Clause。

它们可以支撑 RTC 底层，但不会自动提供 AI Agent、Qwen Realtime 接入、前端状态投影、语音打断、视觉上下文。除非团队目标就是掌握底层 RTC，否则从它们开始会把资源花在非核心处。

这不是它们不好，而是当前对象不允许把底层通信实现误认为完整人工制品。

来源：
- https://github.com/versatica/mediasoup
- https://github.com/meetecho/janus-gateway
- https://github.com/pion/webrtc
- https://github.com/aiortc/aiortc

## 六、按构成性条件映射轮子

下面不是“分层”，而是“职责域到候选手段”的映射。职责域之间有交叉和反馈，不是完全正交自然层。

### 1. 用户设备与权限职责

目标：摄像头、麦克风、扬声器、屏幕共享、设备切换、权限失败处理。

候选手段：

- 浏览器原生 `getUserMedia` / WebRTC。
- LiveKit Client SDK。
- Stream Video SDK / Vision Agents demo。
- 移动端原生 SDK：LiveKit iOS/Android 或 Stream Video SDK。

判断：如果走 Vision Agents，Stream 的前端/edge 体系最顺；如果走长期自控产品，LiveKit 多端 SDK 更稳。

### 2. 实时媒体传输职责

目标：低延迟音频、视频轨道、弱网、回声消除、降噪、房间、断线恢复。

候选手段：

- Stream edge / Vision Agents 默认路径。
- LiveKit server/cloud。
- mediasoup / Janus / Pion / aiortc 作为底层自研选项。

判断：不要从底层开始。Vision Agents 最贴近 Qwen 现成集成；LiveKit 最贴近长期 RTC 产品底座。

### 3. 多模态模型会话职责

目标：听、看、说、打断、转录、上下文、工具调用。

候选手段：

- Qwen3.5-Omni-Flash-Realtime：当前主基座。
- OpenAI Realtime：参考坐标，可替代但不符合你当前阿里云基座前提。
- Gemini Live：参考坐标，可替代但不符合当前基座前提。
- 自托管 Qwen3-Omni：研究方向，不作为第一落地主线。

判断：当前最合理是以 DashScope Qwen Realtime 为主模型会话引擎。

### 4. Agent 编排职责

目标：Agent 生命周期、加入通话、订阅用户轨道、发布 AI 音频、工具调用、状态事件。

候选手段：

- Vision Agents：已有 Qwen Realtime 插件，最直接。
- LiveKit Agents：成熟，但 Qwen adapter 需自写。
- Pipecat：流水线强，但 Qwen Omni Realtime 需自写 service。
- TEN：候选，需进一步核查 Qwen 集成与许可证。

判断：Qwen-first 原型/产品探索用 Vision Agents；RTC 产品化主干可评估 LiveKit + adapter。

### 5. 视觉增强职责

目标：不把所有视觉任务都丢给大模型；对动作、物体、姿态、OCR、商品、屏幕错误做结构化前处理。

候选手段：

- Vision Agents processors。
- YOLO / Ultralytics。
- Roboflow。
- OpenCV。
- MediaPipe。
- OCR 工具。

判断：普通视频通话可以先靠 Qwen 抽帧理解；强场景如健身、维修、商品识别，应加入专用视觉处理器。

### 6. 公共投影职责

目标：让用户知道 AI 正在听/看/说/被打断/断线/无权限/网络差。

候选手段：

- 前端 UI 自定义。
- RTC data channel 传状态事件。
- Agent 事件转发。
- 字幕流：用户转录 + AI 音频转录增量。

判断：这是产品体验核心，不是可有可无装饰。很多开源框架不会替你定义最终 UI 公共契约。

### 7. 工具调用与外部知识职责

目标：联网搜索、业务 API、地理位置、商品库、RAG、客户系统。

候选手段：

- Qwen Realtime Function Calling / WebSearch。
- Agent 框架工具系统。
- 业务后端 MCP/API/RAG。

判断：工具调用必须有权限与成本边界；不要让模型无限搜索或无限调用。

### 8. 观测、成本、治理职责

目标：延迟、首音频时间、会话时长、帧率、token/音频计费、错误、隐私、审计。

候选手段：

- 自建日志与指标。
- 框架事件 hook。
- DashScope 计费与限流监控。
- RTC 质量指标。

判断：这部分决定产品能否承受真实用户，而不是 demo 是否能跑。

## 七、哪些东西不值得花钱“验证”

这里必须明确，避免再次把成本转嫁到错误对象上。

不值得用你的环境和钱重复证明的：

- WebRTC 是否能做实时音视频。
- 浏览器是否能打开摄像头和麦克风。
- Qwen Realtime 官方是否支持实时音频/视觉/语音输出。
- LiveKit 是否是成熟 WebRTC/Agent 基础设施。
- Vision Agents 仓库是否存在 Qwen 插件。
- PyPI 是否发布了 `vision-agents-plugins-qwen`。

这些是公共事实，靠文档、仓库、代码和包索引查证即可。

值得在项目进入实施前测量的，不叫“证明公共事实”，而叫“项目约束测量”：

- 你的 DashScope 中国内地 endpoint、模型名、音色、免费额度、限流是否与选用插件默认配置一致。
- Vision Agents Qwen 插件是否直接支持中国内地百炼 endpoint，还是要改 base URL/session 参数。
- 目标用户网络下首音频延迟、打断延迟、视频帧输入成本和稳定性。
- 浏览器/手机端在真实设备上的回声、扬声器播放、麦克风占用、权限失败体验。
- 甲方业务场景中，1 fps / 2 fps / 事件触发抽帧哪个足以满足视觉理解。
- 工具调用和联网搜索的频率、成本、权限边界。

只有这些问题会影响 `D ∧ S ⊨ R` 是否在你的项目约束下成立。

## 八、真正缺口分类

### 1. 工程胶水缺口

这是最主要的缺口，不是科研缺口。

包括：模型事件转框架事件、RTC 音频轨道转 Qwen PCM、视频轨道抽帧转 JPEG、Qwen 音频输出转 RTC 音轨、字幕事件转前端 UI、打断事件同步前端播放缓冲。

Vision Agents 的 Qwen 插件已经覆盖了其中不少内容。LiveKit 路线则需要自己补。

### 2. 产品公共契约缺口

开源框架不会替你定义用户看到什么。

需要定义：摄像头开启状态、AI 正在看/听、AI 正在回答、用户打断、网络断开、权限失败、模型错误、视频关闭、麦克风静音、搜索/工具调用提示、隐私提示。

这不是前端装饰，而是人工制品公共投影。

### 3. 视觉上下文缺口

头部闭源产品看起来能“记住刚才画面里的东西”，但开源组合里这部分需要自己设计。

Qwen Realtime 有上下文能力，但视频帧不可能无限进入上下文。需要设计：最近帧缓存、视觉摘要、指代对象、关键帧策略、场景状态表、用户追问时如何补帧。

这属于产品/工程设计，不是底层科学未知。

### 4. 成本治理缺口

音频输入、视频输入、音频输出、搜索工具都会计费。抽帧频率和会话时长直接影响成本。

需要把“看得更频繁”与“成本更高、延迟更大、上下文污染更重”放进同一个决策里。默认 1 fps 是合理起点，但强视觉场景可事件触发提高帧率。

### 5. 前沿缺口

真正可能接近前沿的，不是 WebRTC 或普通 Agent，而是：

- 完全本地开源模型达到 OpenAI/Gemini/Qwen 云端 Realtime 的端到端自然语音延迟和稳定性。
- 长时动态视觉上下文中稳定指代、少幻觉、低成本记忆。
- 多模态实时打断中，模型状态、音频播放状态、上下文取消之间的一致性。
- 强视觉任务中，把专用 CV 与全模态 LLM 结合成可靠闭环。

这些需要研究和迭代，但不是当前第一步必须从零发明。

## 九、推荐组合

### 推荐一：Vision Agents + Qwen Realtime 插件

适用条件：你优先想把已有 Qwen3.5-Omni-Flash-Realtime 基座尽快转成“用户摄像头 + 麦克风 + AI 语音回应”的视频 Agent，愿意接受 Stream edge/账号体系作为实时通话承载。

成立理由：

- 已有 Qwen Realtime 插件。
- 已有视频轨道抽帧、音频输入、音频输出、打断处理。
- 插件已经发布 PyPI。
- 对当前 Qwen 基座最短。

主要风险：

- 框架较新。
- 绑定 Stream 路径较强。
- 默认国际 endpoint 与中国内地百炼 endpoint 需要核对。
- 需要检查插件对 `qwen3.5-omni-flash-realtime` 的参数兼容。
- 产品 UI、隐私、成本、工具调用仍需自己定义。

这条路线不是全开源全本地，但它是“开源 Agent 框架 + Qwen 云模型 + 现成 Qwen 插件”的最直接组合。

### 推荐二：LiveKit + 自写 Qwen Realtime Adapter

适用条件：你更看重长期产品化、RTC 控制权、多端 SDK、房间能力、录制、SIP、后续人工接管或企业部署。

成立理由：

- LiveKit 是成熟 RTC/Agent 底座。
- Apache-2.0，社区和商业生态都强。
- 用户端通话产品能力完整。
- 可自托管，也可云服务。

主要成本：

- 当前没有查到官方 Qwen Realtime 插件。
- 需要写 adapter。
- 需要把 Qwen 的 WebSocket 事件与 LiveKit track/data channel 对齐。

这条路线不如 Vision Agents 快，但长期结构更稳。

### 推荐三：Pipecat 作为实验和备选编排层

适用条件：你要比较多模型、多语音服务、多 CV processor，或需要研究 Qwen 端到端与级联 pipeline 的差异。

成立理由：

- Pipeline 抽象强。
- 多供应商支持好。
- 适合服务端语音/多模态实验。

当前限制：

- Qwen 集成不是 Qwen Omni Realtime 视频服务。
- 要达到当前目标仍需自写 service。

### 不推荐作为主线：从 mediasoup / Janus / Pion / aiortc 开始

这些是底层组件。除非目标是自研 RTC 底座，否则会过早把成本放到底层媒体协议，而不是 AI 视频通话体验。

## 十、最终判断

如果把当前任务写成一个清晰命题：

> 在 2026 年现有技术生态中，围绕阿里云百炼 `qwen3.5-omni-flash-realtime`，寻找能搭建 ChatGPT/Gemini Live/豆包式“摄像头 + 麦克风 + 实时视觉语音对话”体验的开源技术方案。

我的回答是：

**存在可落地的开源工程方案，但不存在一个完全开源、全本地、完全等价头部闭源产品的一键成品。当前最贴近你 Qwen 基座的现成开源入口是 Stream Vision Agents + `vision-agents-plugins-qwen`；长期产品化可评估 LiveKit + 自写 Qwen Realtime Adapter；Pipecat 作为多模型流水线实验框架保留；TEN 作为候选观察；底层 WebRTC 项目不作为主线。**

换成 `D ∧ S ⊨ R` 的形式：

- `D`：用户需要消费级视频语音 AI；已有 Qwen Realtime 云模型；头部产品闭源；开源生态存在 RTC/Agent/视频框架；资源有限，不能科研式重造公共技术。
- `R`：用户能开摄像头和麦克风，与 AI 进行低延迟、可打断、视觉上下文连续的自然语音对话。
- `S` 的强候选：Vision Agents + Qwen Realtime 插件 + Stream edge + 前端公共投影 + 成本/隐私/工具治理。
- `S` 的长期候选：LiveKit + Qwen Realtime adapter + 自定义产品前端 + 工具/治理层。

这不是发明灯泡。真正需要创造的是“把已有轮子放到正确职责边界里，并补齐目标人工制品缺失的契约”。

如果后续继续推进，我认为下一份产物不应该是跑 demo，而应该是两份更具体的东西：

1. **目标体验规约**：逐条定义用户旅程、状态投影、打断语义、视觉上下文、失败语义、隐私与成本边界。
2. **Vision Agents/Qwen 与 LiveKit/Qwen 两条候选架构的接口契约对比**：看哪条更满足甲方的产品约束、部署约束、成本约束和时间约束。

只有当这两份契约清楚以后，环境测量才有资格发生。届时要测的不是“Qwen 是否真的能工作”，而是“在我们选定的 `D` 与 `S` 下，目标 `R` 是否可被推出”。

# 新书设计：把“类似 GPT/Gemini/豆包的视频通话体验”写成可判定的公共契约

我已经完整阅读了 07 号书《结构化书写约定标本》，也补读了 semantic-writing-discipline 的对象模型、话语构造和审查清单。07 号书给我的关键启发不是某个 AsciiDoc 写法本身，而是它把一本书当成一种可维护、可引用、可投影的人工制品：标题有稳定 ID，章节有 role 身份，引用有 rel 关系，术语表和索引服务读者与工具链，附录承载速查和结构化补充。它不是散文，不是帮助页，也不是一次性说明书。

你现在要的这本书，也应该是这种对象。它不是技术选型报告，不是 Qwen 接入指南，不是前端设计稿，不是项目计划，不是某个供应商的对标文档。它应当是一本**体验规约与符合性判定书**：把“消费级实时全模态 AI 视频语音通话体验”正面定义出来，使不同消费者都能用同一套词、同一套流、同一套黑盒判据判断一个实现到底是不是这个东西。

我建议这本书的工作名为：

**《实时全模态视频通话体验规约与符合性判定》**

也可以用稍短的标题：

**《实时全模态视频通话体验规约》**

我更推荐第一个，因为“符合性判定”四个字直接表达它和 TCK 类比中的核心效果：不是只描述愿望，而是让一个声称实现该体验的产品接受判定。它不会变成 Java TCK 那种具体测试套件，但它会承担类似的上游角色：定义对象、定义承诺、定义观察面、定义基本流、定义分支流、定义不可降级条件、定义符合性声明需要满足什么。

## 一、这本书的对象身份

这本书的对象是：**消费级实时全模态 AI 视频语音通话体验**。

这个对象的用户表面是：用户打开摄像头与麦克风，AI 在同一会话中接收用户语音和视觉现场，理解用户语言、画面内容、指代关系、上下文变化，并以自然语音回应；用户可以打断、切换话题、关闭或恢复摄像头/麦克风；系统通过可观察状态让用户知道 AI 正在听、正在看、正在回应、已被打断、遇到错误或无法继续。

这本书不定义内部实现。它不要求某个产品必须使用 Qwen、OpenAI、Gemini、LiveKit、Vision Agents、WebRTC、SFU、STT、TTS、VLM 或某个具体框架。那些属于求解空间。对这本书来说，一个实现只要黑盒行为满足规约，就有资格被称为符合本书定义的实时全模态视频通话体验；如果黑盒行为不满足规约，即使内部技术很先进，也不能声称符合。

这本书也不定义商业形态。它不限定移动端、网页端、桌面端，也不限定 C 端助手、客服、教育、维修、健身或导购场景。它定义的是体验对象的构成性条件和符合性判据。具体业务产品可以在这个对象上叠加自己的业务职责。

因此，这本书的本体位置应当是：

- 它高于技术架构书。
- 它低于纯粹愿景宣言。
- 它属于需求工程中的可判定体验规约。
- 它使用陈述语气记录已知事实和边界。
- 它使用愿望语气定义目标状态。
- 它使用规范性语句定义符合性要求。
- 它使用黑盒场景定义可观察验收。

这本书的核心不是“我们想做一个像谁的产品”，而是“当一个产品声称自己提供实时全模态 AI 视频语音通话体验时，它必须在用户可观察层面满足哪些承诺”。

## 二、这本书服务的 actor

这本书的消费者必须正面定义。读者不是模糊的“可能包括谁”，而是以下几类 actor。每一类 actor 都有稳定用途；书的结构必须让他们各自拿到需要的信息。

### 1. 需求提出者 / 甲方代表

他们需要这本书判断：自己口中的“像 ChatGPT/Gemini/豆包那种视频通话 AI”到底是什么。他们不需要知道模型 API、RTC 框架或插件细节。他们需要一套可读的体验对象语言，用来表达目标、审查方案、验收交付。

这本书给他们的价值是：把模糊愿望变成可讨论、可确认、可验收的体验规约。以后他们不需要反复说“就像某某产品那样”，而是可以指向本书中的章节、基本流、分支流和符合性条款。

### 2. 产品负责人 / 交互设计者

他们需要这本书判断：用户从进入通话、授权设备、开始说话、展示画面、等待回应、打断、关闭摄像头、恢复会话，到结束会话，每一步需要什么可见状态、动作入口和反馈。

这本书给他们的价值是：提供用户旅程、动态语义、公共投影和不可降级体验条件。它能防止产品设计只画一个“通话界面”，却遗漏打断、权限失败、视觉指代、字幕、隐私提示、错误恢复这些构成性体验。

### 3. 架构师 / 技术负责人

他们需要这本书判断：技术方案是否覆盖目标对象的所有职责域。架构师不在这本书里找具体实现代码，而是用它建立责任分解：设备输入、实时媒体、模型会话、视觉上下文、语音输出、状态投影、工具调用、治理。

这本书给他们的价值是：把技术选型从“哪个框架热门”拉回“哪些黑盒承诺必须成立”。它也能帮助架构师识别哪些是构成性条件，哪些只是可替换手段。

### 4. 开发者 / 实现团队

他们需要这本书判断：实现产物必须对外呈现哪些行为。开发者不应该从这本书里寻找某个库的用法，但应当能从本书中知道输入、事件、状态、错误、输出和用户可见结果的契约。

这本书给他们的价值是：减少“做出来了但不是那个东西”的返工。开发者可以把本书中的符合性条款转成实现任务、接口契约和自动化测试用例。

### 5. 测试工程师 / 验收负责人

他们需要这本书判断：一个实现是否符合规约。测试工程师需要基本流、分支流、场景矩阵、黑盒可观察结果、失败语义和判定标准。

这本书给他们的价值是：提供接近 TCK 效果的判定依据。它不是自动化测试套件，但它会把测试断言写清楚：给定什么前置状态，用户执行什么动作，系统应当出现什么可观察结果。

### 6. 采购 / 评审 / 合作方审查者

他们需要这本书判断：一个供应商声称“我们也做了实时视频 AI 对话”是否成立。供应商可能会展示 demo、视频、PPT 或 API 能力，但这些都可能只覆盖局部。

这本书给他们的价值是：提供供应商符合性声明的审查表。一个实现若只支持拍照问答、只支持语音通话、不能打断、无法说明视觉上下文、没有权限失败语义，就不能被误认为符合本书对象。

### 7. 安全、隐私与运营负责人

他们需要这本书判断：摄像头、麦克风、语音、画面、转录、日志、搜索和工具调用如何影响用户权益、数据边界和运营成本。

这本书给他们的价值是：把隐私提示、采集状态、数据保存、成本控制、错误治理、用户退出权写成体验规约的一部分，而不是后补条款。

这七类 actor 都是清晰消费者。书的章节应当均衡服务他们：不能只服务架构师，不能只服务测试，不能只服务产品，也不能只服务甲方愿望。

## 三、这本书的效果目标

这本书完成后，应当产生四种效果。

第一，**命名效果**。以后团队不再用“类似 GPT 那个”“类似 Gemini Live 那个”“像豆包视频通话那个”作为主要表达，而是使用本书定义的对象名、章节名和术语。

第二，**规约效果**。团队可以知道一个实现必须满足哪些条件才是本书所定义的对象。这里的条件不是内部技术条件，而是体验构成条件和可观察行为条件。

第三，**验收效果**。测试与验收可以把本书中的基本流、分支流和判定条款转成黑盒检查。供应商或内部团队声称实现时，可以按本书逐条对照。

第四，**边界效果**。本书会明确什么不属于当前对象。例如内部模型架构、具体 RTC 框架、具体云厂商、营销文案、数字人形象、业务知识库，不是这本书的核心定义层。它们可以成为某个实现或某个业务产品的手段，但不能替代对象本身。

这里要吸收 JCP/TCK 的效果，但不照搬形式。JCP 对 TCK 的说明强调：TCK 服务于实现与规格之间的兼容性判断；符合性测试是黑盒的，测试指定内容而不是实现内部；高质量 TCK 需要文档、自动运行机制、覆盖说明、申诉机制；兼容性要求还包括测试之外必须满足的要求。我们这本书不是 Java TCK，不写可执行测试套件，但要继承这个思想：**把符合性判断从供应商自说自话中拿出来，放到公开规约、可观察结果和可争议条款里。**

## 四、这本书不是什么

这些排除不是为了反复否定相邻对象，而是为了防止高成本误读。

这本书不是技术选型报告。它不推荐 Vision Agents、LiveKit、Pipecat、TEN 或任何模型服务。

这本书不是实现计划。它不安排第一阶段、第二阶段，也不规定开发顺序。

这本书不是用户手册。它不教最终用户如何点击按钮，而是定义一个符合实现应当暴露哪些动作和状态。

这本书不是供应商对比报告。它可以被用来评估供应商，但它不以某个供应商为对象。

这本书不是自动化测试代码。它提供测试断言和黑盒场景，后续可以转成自动化测试，但书本身不是测试执行器。

这本书不是模型能力宣传。它不因为某个模型支持视频、音频、Function Calling 就判定产品体验成立。模型能力只是实现手段。

这本书不是纯 UX 灵感集。它写的是可判定的体验规约，不是设计氛围、文案风格或视觉情绪板。

## 五、规范性语言纪律

这本书需要有明确的规范性词汇。RFC 2119 的价值在于把 MUST、SHOULD、MAY 这类词的要求强度稳定下来。中文书不必机械使用英文大写，但应当定义中文规范词。

我建议使用四类词：

**必须**：符合实现的绝对要求。缺少该项时，不能声称符合本书定义的体验。

**不得**：符合实现的绝对禁止。出现该行为时，即使其他部分完成，也构成不符合。

**应当**：强推荐要求。实现可以在有明确理由时偏离，但偏离必须被记录，并说明对用户体验、验收和风险的影响。

**可以**：可选能力。实现包含它时，必须不破坏必须项；实现不包含它时，不影响基本符合性。

这套词要非常克制。不能把所有愿望都写成“必须”。RFC 2119 也提醒规范性词汇应谨慎使用，尤其只在互操作、风险或必要行为上使用。对本书来说，“必须”只放在对象构成性条件上；“应当”放在强体验质量要求上；“可以”放在场景增强或可替换表现上。

例如：

- “系统必须在用户授权摄像头前不采集视频。”这是隐私与权限构成性要求。
- “系统必须在摄像头开启状态下给出可见采集状态。”这是公共投影要求。
- “系统应当在用户打断后停止继续播放被打断回答。”这是体验强要求，很多实现若做不到会严重破坏对象，但具体停止方式可能依赖平台。
- “系统可以显示实时字幕。”如果我们把字幕定义为辅助投影而非构成性条件，它就是可以；如果后续决定字幕是验收必需，就改为必须。

书中每条规范性语句都应有稳定 ID。后续测试、评审、供应商答复、issue、变更记录都引用 ID，而不是引用自然段位置。

这一部回答“用户和系统如何发生交互”。

1. `010-session-lifecycle.adoc`：会话生命周期。

   - 用户切换话题。
   - 用户回到先前画面对象。
   - 系统如何表现“不知道”“看不清”“需要重新展示”。

这一部主要服务产品、交互、开发、测试。它应当包含大量黑盒场景，但不写代码。

### Part 300：公共投影与可观察结果

这一部回答“用户和外部评审者能看到什么”。

建议章节：

1. `010-observable-state-model.adoc`：可观察状态模型。
   - 未连接、等待授权、摄像头开启、麦克风开启、正在听、正在看、正在回应、被打断、网络异常、已结束。

2. `020-user-visible-feedback.adoc`：用户可见反馈。
   - 状态图标、字幕、波形、说话状态、摄像头采集提示、错误提示。
   - 不规定视觉设计风格，只规定必须表达的信息。

3. `030-speech-output-contract.adoc`：语音输出契约。
   - AI 语音回应的开始、继续、结束、取消。
   - 输出音频不得与用户新输入形成不可理解的重叠。

4. `040-transcript-and-caption-contract.adoc`：转录与字幕契约。
   - 用户语音转录是否必须显示，需要在本书中决策。
   - AI 回复字幕是辅助投影还是必需投影，也要明确。

5. `050-error-projection.adoc`：错误投影。
   - 权限失败、网络断开、模型不可用、视频不可用、音频不可用、工具调用失败。
   - 错误不能只写日志，必须有用户可见语义。

6. `060-privacy-projection.adoc`：隐私投影。
   - 用户必须知道摄像头和麦克风何时被采集。
   - 用户必须能停止采集。
   - 如果保存转录、图片帧或会话记录，必须有明确提示。

这一部是书的验收核心之一。Gherkin 的一个重要思想是 Then 应该检查可观察输出，而不是内部数据库状态。这里也一样：符合性判定应优先看用户或外部系统能观察到的结果。

### Part 400：符合性断言与黑盒场景

这一部最接近你说的 TCK 效果。它不是测试代码，但它把测试断言写成稳定条款。

建议章节：

1. `010-assertion-form.adoc`：断言写法。
   - 每条断言包含 ID、前置条件、用户动作、期望结果、适用范围、规范强度。
   - 可以借鉴 Given/When/Then，但不必完全采用英文关键字。

2. `020-core-flow-assertions.adoc`：核心流断言。
   - 进入会话。
   - 授权设备。
   - 语音问答。
   - 视觉问答。
   - 语音 + 视觉联动。
   - 结束会话。

3. `030-branch-flow-assertions.adoc`：分支流断言。
   - 权限拒绝。
   - 摄像头关闭。
   - 麦克风静音。
   - 网络异常。
   - 模型超时。
   - 用户中途打断。

4. `040-visual-context-assertions.adoc`：视觉上下文断言。
   - 当前画面对象识别。
   - 指代承接。
   - 画面变化后的上下文更新。
   - 看不清时的诚实反馈。

5. `050-interruption-assertions.adoc`：打断断言。
   - 用户说话触发打断。
   - 当前音频输出停止。
   - 被取消回答不继续输出。
   - 新问题得到新回答。

6. `060-nonconformance-cases.adoc`：不符合案例。
   - 只支持拍照上传。
   - 只支持语音不支持视觉。
   - 只能播完不能打断。
   - 摄像头开启无可见状态。
   - 权限失败无解释。
   - AI 声称看到了实际未提供的画面。

7. `070-coverage-map.adoc`：覆盖映射。
   - 每条构成性条件对应哪些断言。
   - 每条断言覆盖哪个用户流或公共投影。
   - 明确哪些要求暂不被断言覆盖。

JCP/TCK 文档强调 TCK 需要 coverage document，说明覆盖标准和覆盖质量。我们这本书也应该有覆盖映射，但它不是代码覆盖率，而是体验构成条件到黑盒断言的覆盖。

### Part 500：质量、治理与边界条件

这一部回答“满足核心体验后，还必须怎样不伤害用户、成本和运营”。

建议章节：

1. `010-latency-and-responsiveness.adoc`：延迟与响应性。
   - 不给绝对统一数值，除非项目后续确认；但定义哪些延迟是用户可感知指标：首个语音反馈、打断停止、视觉追问回应。

2. `020-continuity-and-memory-boundary.adoc`：连续性与记忆边界。
   - 系统应当承接当前会话内的相关视觉与语言上下文。
   - 系统不得暗示无限记忆。
   - 看不清或上下文缺失时必须诚实表达。

3. `030-cost-and-resource-boundary.adoc`：成本与资源边界。
   - 会话时长、视频帧、语音输出、搜索、工具调用会消耗资源。
   - 用户体验规约需要允许实现控制采集频率，但不得破坏核心承诺。

4. `040-tool-use-boundary.adoc`：工具调用边界。
   - AI 调用搜索、位置、业务 API 时，用户是否可见、是否需要授权、失败时如何表达。

5. `050-safety-and-privacy-boundary.adoc`：安全与隐私边界。
   - 摄像头、麦克风、屏幕共享属于高敏输入。
   - 采集、保存、转发和删除必须有明确公共语义。

6. `060-accessibility-and-environment.adoc`：可访问性与环境适应。
   - 嘈杂环境、弱网、无摄像头、无麦克风、听障用户、字幕需求。
   - 这些不应无限扩大对象，但应当作为符合性或扩展符合性条件记录。

这一部主要服务安全、运营、产品和验收。

## 七、章节 role 与 rel 约定

参考 07 号书，新书也应该定义 role 和 rel。它们不是排版装饰，而是让书可维护、可查询、可投影。

### 建议 role

`.concept`：概念定义，例如“实时全模态视频通话体验”。

`.domain`：领域事实，例如设备、网络、权限、用户环境。

`.goal`：目标或愿望，例如用户希望 AI 看见现场并回应。

`.requirement`：规范性要求，例如系统必须在摄像头采集时显示状态。

`.contract`：公共契约，例如语音输出契约、错误投影契约。

`.flow`：用户基本流或分支流。

`.state`：状态定义，例如正在听、正在看、正在回应。

`.assertion`：符合性断言。

`.criterion`：验收判据。

`.nonconformance`：不符合案例。

`.boundary`：边界规则。

`.term`：术语定义。

`.template`：模板。

`.appendix`：附录材料。

每个 role 必须在术语表或附录中定义。一个 role 在全书中只能表达一种身份。

### 建议 rel

`defines`：当前标题定义目标对象。

`depends-on`：当前判断依赖目标事实或概念。

`satisfies`：当前要求服务目标目标或构成性条件。

`verifies`：当前断言验证目标要求。

`constrains`：当前规则约束目标对象范围。

`refines`：当前章节细化目标章节。

`observes`：当前公共投影观察目标状态或事件。

`excludes`：当前边界排除高成本误读对象。

`extends`：当前能力扩展核心对象但不是核心符合性条件。

这些关系能让后续维护者从“构成性条件”追踪到“要求”，再追踪到“断言”，再追踪到“验收场景”。这就是 07 号书结构化表面的真正价值。

## 八、断言写法

这本书需要一种稳定的黑盒断言表面。可以吸收 Gherkin 的思想：初始上下文、事件、可观察结果。Cucumber/Gherkin 官方参考强调：Given 描述系统初始上下文，When 描述事件或动作，Then 描述预期结果；Then 应检查用户或外部系统可观察输出，而不是内部数据库这类隐藏状态。这个思想非常适合本书。

我建议使用中文表格断言，而不是直接写英文 Gherkin。格式如下：

[cols="1,3", options="header"]
|===
|字段 |含义

|断言 ID
|稳定引用地址，例如 `ASSERT-CAM-001`。

|规范强度
|必须 / 不得 / 应当 / 可以。

|前置条件
|系统和用户处于什么可观察状态。

|用户动作
|用户执行什么动作，或外部事件发生什么。

|期望结果
|用户或外部评审者能观察到什么结果。

|不符合表现
|哪些行为构成失败。

|覆盖要求
|该断言验证哪些 requirement ID。
|===

例如：

断言 ID：`ASSERT-CAM-STATE-001`

规范强度：必须

前置条件：用户已进入 AI 视频语音通话会话，摄像头权限已授予。

用户动作：用户开启摄像头。

期望结果：系统在用户界面中显示摄像头正在采集的可见状态；用户能执行关闭摄像头动作；AI 后续回答可以基于摄像头画面内容。

不符合表现：摄像头已采集但界面无任何状态；用户无法关闭摄像头；AI 声称看不到画面但系统仍显示摄像头开启；系统采集画面但未获得用户授权。

覆盖要求：`REQ-PRIVACY-001`、`REQ-VISION-001`、`REQ-PROJECTION-001`。

这种断言不是代码，但已经足以让测试、验收、供应商审查和后续自动化测试工作接入。

## 九、符合性等级

这本书应该定义符合性等级，但不要过度复杂。我建议四类：

**核心符合**：实现满足所有“必须”和“不得”要求，核心流断言全部通过。可以声称符合本书定义的核心实时全模态视频通话体验。

**条件符合**：实现满足核心要求，但存在明确环境或范围限制，例如仅支持 Web 端、仅支持单人会话、仅支持摄像头不支持屏幕共享。限制必须在符合性声明中公开。

**扩展符合**：实现满足核心符合，并额外满足扩展能力，例如屏幕共享、多人参与、业务工具调用、视觉检测增强、数字人形象。

**不符合**：实现缺少构成性条件，或违反“不得”要求，或核心流断言失败。

这里要避免让“扩展能力”污染核心定义。比如数字人形象、屏幕共享、专用业务工具、长期记忆、多人协作，可能很有价值，但不能一开始就变成核心符合项。它们应被放入扩展符合。

## 十四、我对这本书写作方法的判断

这本书必须使用对象语言。不要写“我们先做”“后续支持”“第一版可以”“未来考虑”这类过程语言。过程可以存在于项目计划，不进入核心定义层。

这本书必须区分：

- 事实：用户设备、网络、权限、摄像头、麦克风、现有产品表面事实。
- 愿望：用户希望 AI 听见、看见、理解并回应。
- 要求：符合实现必须满足的行为。
- 契约：系统对用户和外部观察者的可依赖承诺。
- 断言：可用于黑盒判定的场景。
- 扩展：有价值但不构成核心符合的能力。

这本书不应该用产品名替代定义。ChatGPT、Gemini Live、豆包可以出现在参考坐标或动机背景中，但核心章节应当逐渐消除对这些名字的依赖。真正完成后，一个没有用过这些产品的人，也应能读懂本书定义的体验对象。

## 一、关于断言写法：断言是标题单位，不是表格里的字面量编号

我修正先前的断言形式。

之前我提出过类似 `ASSERT-CAM-STATE-001` 的表格字段。这个形式有一个问题：它把断言 ID 当成字面量字段，而不是书稿结构的一部分。这样做虽然能被人眼识别，但不够符合 07 号书里强调的稳定 ID、role 和 xref 结构。它也会削弱下游维护：如果某个断言改名、移动章节、被拆分或被合并，字面量 ID 容易成为孤立字符串。

正确方式是：**每一条符合性断言都写成一个小一级标题**。

例如在 `040-visual-context-assertions.adoc` 中，不写：

```asciidoc
|断言 ID |ASSERT-VISUAL-REFERENCE-001
```

而写：

```asciidoc
[#assert-current-visible-object.answerable.assertion, normative=must]
=== 当前可见对象可被询问

[horizontal]
规范强度:: 必须。
前置条件:: 用户已进入会话；摄像头处于开启状态；画面中存在一个用户可见且可被常识性描述的对象。
用户动作:: 用户以语音询问“这个是什么”或等价表达，并以摄像头画面作为指代来源。
期望结果:: 系统以语音回应，并使回应内容与当前画面对象形成可理解关联。
不符合表现:: 系统只要求用户拍照上传；系统只基于语音文本回答；系统声称无法接收画面；系统对不存在于画面的对象作确定描述。
覆盖要求:: xref:req-visual-current-frame[当前视觉现场参与回答, rel=verifies]。
```

这里有几个关键点：

- `[#assert-current-visible-object.answerable.assertion]` 是稳定标题 ID 和 role。
- `=== 当前可见对象可被询问` 是读者可见标题。
- `normative=must` 是标题附加字段，表示规范强度。
- 其他章节引用这条断言时，写 `xref:assert-current-visible-object[当前可见对象可被询问]`。
- 覆盖关系通过 `rel=verifies` 指向 requirement 标题。
- requirement 本身也是标题单位，不是字面量编号。

这使书稿结构变成可导航图：

```text
目标 / 构成性条件
  -> requirement 标题
    -> assertion 标题
      -> 场景前置条件 / 用户动作 / 期望结果 / 不符合表现
```

这也意味着书中不需要手动维护一堆“REQ-001”“ASSERT-001”字面量。稳定 ID 仍然存在，但它藏在标题 attrlist 中，由 xref 使用；读者看到的是自然标题，工具链看到的是稳定结构。

我会把这种写法作为本书的断言公共表面。

## 二、关于“推荐”“初稿”“首批”：我撤回这些表达

如果我说“首批”，就容易暗示先写一点再补一点。对这本书来说，这会造成很坏的路径依赖：先写出来的部分会定义后续章节的语义空间。如果最先写的部分不完整、不严谨，后面会被迫在错误结构上补洞。

## 三、关于书的篇幅与信息密度：这不是骨架书

你指出“每一章随便三四十行就是糊骨架”，这个判断成立。

07 号书是结构样本，它可以短，因为它的对象是“展示结构化书写约定标本”。但我们这本书的对象不是样本，而是一个高层体验规约与符合性判定。它要服务甲方、产品、架构、开发、测试、供应商评审、安全运营这些 actor。它必须降低这些读者面对“类似 GPT/Gemini/豆包那种东西”时的条件熵。

这意味着每个 chapter 的物理篇幅和语义负载都要足够。不能只列标题。不能只写“系统应支持打断”。打断章节必须说明：

- 打断发生在哪些状态下。
- 用户以什么动作触发打断。
- AI 音频输出如何停止。
- 视觉上下文是否保留。
- 被取消回答是否还能继续影响后续。
- 前端如何投影打断状态。
- 哪些行为构成不符合。
- 哪些断言验证打断。
- 哪些相邻概念不是打断，例如暂停播放、静音、网络中断。

这才叫把对象写出来。

我对书稿规模的判断是：**整本书物理篇幅应当超过 2 万 token**。这不是为了长，而是因为对象复杂度需要这个表达空间。每个 part 不能只是导语，每个 chapter 不能只是标题解释。章节要包含对象定义、边界、规则、场景、断言、反例、交叉引用。

我会用一个内部标准检查每章的信息密度：

1. 这个 chapter 是否定义了一个独立对象或职责面？
2. 它是否说明了该对象在整本书中的位置？
3. 它是否给出了必要术语？
4. 它是否把用户可观察行为写清楚？
5. 它是否给出了符合与不符合的边界？
6. 它是否能被至少一个 actor 直接消费？
7. 它是否有交叉引用连接到上游 requirement 或下游 assertion？
8. 删除该章后，整本书是否失去判定能力？

如果某章不能通过这些问题，就不是完整 chapter。

## 四、书的逻辑依赖关系

写这本书不能按文件列表随意推进。它有依赖图。

### 1. 对象身份先于所有要求

`artifact-identity` 必须先写。因为如果对象身份不稳定，后面的 flow、state、assertion 都没有锚点。

它必须回答：

- 本书定义的体验对象是什么。
- 它的用户表面是什么。
- 它的构成性条件是什么。
- 它和语音助手、图片问答、视频会议、数字人、技术 demo 的差别在哪里。
- 它的符合实现需要承担什么公共承诺。

### 2. 领域事实先于目标愿望

`domain-assumptions` 必须在目标章节之前写清。因为需求工程里 `D ∧ S ⊨ R` 的 D 不是装饰。没有 D，愿望会悬空。

这里要写的是：

- 用户设备事实。
- 摄像头、麦克风、扬声器权限事实。
- 网络事实。
- 实时模型服务事实。
- 视觉输入和语音输入的敏感性事实。
- 消费级用户不会阅读技术说明的事实。
- 失败和延迟不可避免的事实。

这章必须用陈述语气。

### 3. 目标愿望先于系统边界

`stakeholder-goals` 写 optative mood：用户希望 AI 听见、看见、回应、承接、打断、说明失败。

目标写完以后，`system-boundary` 才能判断哪些进入当前对象，哪些属于扩展，哪些属于相邻人工制品。

### 4. 边界先于符合性模型

如果边界没写清，符合性就会膨胀。比如屏幕共享、多人会话、数字人形象、长期记忆、业务 RAG 是否是核心符合项，必须在边界章节里归档。

`conformance-model` 依赖边界章节。它定义核心符合、条件符合、扩展符合、不符合、偏离声明。

### 5. 状态模型先于流程

`observable-state-model` 或者会话生命周期里的状态定义，必须在具体流之前稳定。否则“正在听”“正在看”“正在回应”“被打断”会在不同章节漂移。

我会把状态模型前置到用户旅程部分的第一批依赖中。会话生命周期和状态模型可以互相引用，但状态名必须在第一次出现处定义。

### 6. 核心流先于分支流

基本对话流、视觉指代流、打断流必须在分支流之前写清。分支流是核心流在异常事实下的变体。如果核心流不清楚，分支流只会变成错误列表。

### 7. 公共投影先于黑盒断言

断言检查的是可观察结果。如果公共投影没定义，断言会检查不到对象。例如“AI 被打断”必须有可观察投影，否则测试只能猜内部状态。

因此断言章节依赖公共投影章节。

### 8. 断言覆盖图最后形成

`coverage-map` 必须在 requirement 与 assertion 都写完后生成。它不是提前设想，而是全书结构的覆盖投影。

这个依赖关系决定写作顺序。不是文件顺序，也不是目录漂亮，而是语义上谁依赖谁。

## 六、每个 part 的写作负载

### Part 100：对象、边界与符合性模型

这一部是整本书的本体根。它必须写得足够完整。预计它会是全书最重要的部分之一。

它不是介绍，不是背景，不是“为什么要做”。它负责建立：本书定义的对象是什么，世界事实是什么，愿望是什么，边界在哪里，符合性如何判定。

如果这一部不完整，后面所有章节都会漂移。

这一部至少要覆盖：

- 对象身份。
- 构成性条件。
- 领域事实。
- stakeholder goals。
- 当前对象与相邻对象的边界。
- 核心符合、条件符合、扩展符合、不符合。
- 偏离声明的义务。

### Part 200：动态体验

这一部是用户行为世界。它不是 UI 说明，而是动态语义。

它要写：

- 会话如何开始和结束。
- 系统有哪些状态。
- 用户授权如何改变状态。
- 用户语音如何进入会话。
- 用户视觉现场如何进入会话。
- 用户指代如何和视觉现场绑定。
- 用户打断如何改变回答和播放。
- 模态关闭和恢复如何影响体验。
- 话题和上下文如何连续或中断。

这一部要避免内部实现词。比如不写“WebSocket 发送帧”，而写“符合实现必须使当前视觉现场在回答中可被使用”。

### Part 300：公共投影

这一部是外部可见世界。它负责把状态和行为变成用户可理解的表面。

它要写：

- 用户如何知道摄像头正在采集。
- 用户如何知道麦克风正在采集。
- 用户如何知道 AI 正在听、正在看、正在回答。
- 用户如何知道回答被打断。
- 用户如何看到错误。
- 用户如何停止采集。
- 用户如何知道工具调用、搜索、位置或业务数据是否参与。
- 用户如何理解字幕、转录和语音输出之间的关系。

这一部对测试非常重要，因为黑盒断言只能检查可观察结果。

### Part 400：符合性断言

这一部承担 TCK 类效果。

它不写代码，但每条断言都要像可转换成测试一样稳定。每条断言作为标题单位存在。每条断言包含前置条件、用户动作、期望结果、不符合表现和覆盖要求。

这一部会是物理篇幅较大的部分，因为断言不能压缩成一两句。每个核心流程都要有多个断言。

### Part 500：质量、治理与符合性声明

这一部防止“功能能跑但产品不可用”。它处理延迟、成本、隐私、安全、可访问性、供应商声明。

它必须把治理写成体验规约的一部分，而不是后端运维附录。

例如成本边界不是“后台要省钱”，而是用户体验上不能因为成本策略悄悄降级成假实时；如果实现降低视觉帧率，它必须不破坏核心视觉承诺；如果破坏，就必须在符合性声明中说明范围。

## 七、每个 chapter 的最小完整性

我需要给自己一个写作约束：每章不是“有标题就算存在”。每个 chapter 至少包含以下结构，除非该章是纯术语表或纯附录：

1. **对象定义**：本章讨论的对象是什么。
2. **本章边界**：本章不处理什么，避免相邻对象污染。
3. **规范性要求**：符合实现必须、不得、应当或可以做什么。
4. **动态语义或状态影响**：用户动作发生后系统状态如何变化。
5. **公共投影**：用户或外部观察者如何看见结果。
6. **不符合表现**：哪些行为不能被接受。
7. **交叉引用**：本章依赖哪些上游章节，验证哪些下游断言，或约束哪些公共投影。
8. **术语进入**：本章引入的新术语必须进入 glossary 或本章内明确定义。

如果某章只写了抽象说明，没有规范性要求、没有不符合表现、没有投影或交叉引用，它就不能称为完成。

## 九、每批写完后的熵值检查

你要求我检查“熵值有没有被降下去”。我会把它落实成检查问题。

每批完成后，我会读新写章节，并问：

1. 读者原本不确定的对象是什么？
2. 本批文字消除了哪些不确定性？
3. 哪些判断仍然依赖作者脑内上下文？
4. 是否出现“这个”“这样”“类似”“实时”“智能”“自然”这类高熵词但没有定义？
5. 是否把技术手段误写成对象定义？
6. 是否把愿望误写成要求？
7. 是否把要求写成无法观察的内部状态？
8. 是否出现过程语言，例如“后续”“第一版”“先做”“推荐”？
9. 是否有章节只给了标题，没有给判定标准？
10. 是否有 actor 读完后仍不知道如何使用本章？

如果答案暴露问题，我会在同一批次内修正，而不是留给你发现。

这不是数学上精确计算 Shannon entropy，而是以信息论为约束的写作审查：读者的不确定性必须降低，不能靠作者和读者心照不宣。

## 十、关于章节语言的形式

这本书应使用稳态对象语言。

不写：

- “我们可以先支持摄像头，再支持屏幕共享。”
- “第一版默认 1 fps。”
- “推荐展示一个状态图标。”
- “后续可以接 Qwen。”
- “类似豆包那样。”

改写为：

- “符合实现必须在摄像头采集发生时提供用户可见采集状态。”
- “视觉现场参与回答时，系统必须存在将当前视觉现场纳入会话的机制；该机制的内部采样策略不属于本书核心定义。”
- “屏幕共享属于扩展视觉输入能力；未声明扩展符合的实现不因缺少屏幕共享而构成核心不符合。”
- “当系统不能使用当前视觉现场时，必须以用户可理解方式表达该限制。”

这就是对象语言和过程语言的区别。

## 十一、关于技术细节的边界

这本书不写具体技术接口，但并不等于无视技术现实。

它会写：

- 系统必须能接收用户语音输入。
- 系统必须能接收当前视觉现场或明确声明视觉输入不可用。
- 系统必须以自然语音回应。
- 系统必须让用户观察采集状态。
- 系统必须处理打断。
- 系统必须给出错误语义。

它不写：

- 必须使用 WebRTC。
- 必须使用 Qwen。
- 必须使用 Vision Agents。
- 必须用某个采样率。
- 必须每秒传几帧。
- 必须用某个模型事件名。

但它可以在边界章节说明：内部实现可以使用实时模型、RTC、抽帧、音频缓冲、工具调用等手段；这些手段不构成本书的符合性条件，除非它们的行为投影到用户可观察契约上。

这样可以保持书的高层规约身份，同时不写成空泛愿景。

## 十二、关于参考产品的处理

ChatGPT、Gemini Live、豆包不应该在核心章节里作为定义来源。它们可以出现在：

- 前言中的背景说明。

## 三、这本书的结构不应是机械 MECE，而应是论述递进

MECE 是有价值的，但不能机械滥用。一本书不是表格分类器。它需要结构清楚，也需要阅读顺序自然。

这本书的结构应当采用递进关系：

**对象定义 → 体验发生 → 可观察承诺 → 符合性判定 → 治理边界。**

这五个部分不是互相完全隔离的分类盒子，而是读者理解对象的路径。

- 没有对象定义，后面的体验流没有主题。
- 没有体验发生，公共投影没有状态来源。
- 没有公共投影，符合性判定没有观察面。
- 没有符合性判定，书只能表达愿望，不能产生 TCK 类效果。
- 没有治理边界，体验会被成本、隐私、安全和扩展能力污染。

所以，书的结构应该像这样：

1. 第一部建立对象。
2. 第二部展开体验。
3. 第三部定义可观察承诺。
4. 第四部给出符合性判定。
5. 第五部处理治理与扩展边界。

这不是“每章固定栏目”的结构，而是一本非虚构规约书的论述结构。

---

## 四、章节应像章节，而不是模板表单

一个 chapter 应该有自己的内部叙述形态。不同章的写法可以不同。

例如《体验对象》这一章，应当像定义性论文章节：先给出对象名，再说明用户表面，再解释构成性条件，再用反例澄清相邻对象。它不需要一个显式栏目叫“本章不处理什么”。如果要排除“拍照问答”“普通语音助手”，可以自然地写在对象边界的论述里。

《会话生命周期》这一章，应当像状态与过程的说明：从进入会话写到结束会话，说明状态如何转换、用户何时能观察变化、哪些状态不能互相冒充。它可以包含状态表，但表格服务论述，不取代论述。

《视觉指代》这一章，应当像概念和场景结合的章节：解释“这个”“刚才那个”“屏幕上这个错误”为什么是实时视觉体验的核心，然后展开几个可观察场景，说明符合实现如何承接视觉现场，不符合实现如何暴露问题。

《打断》这一章，应当像动态语义章节：说明打断不是暂停按钮、不是静音、不是网络中断，而是用户新输入改变当前回答命运的交互现象。它要把音频输出、模型回答、前端投影和后续上下文的关系讲清楚。

《符合性断言》这一部可以更像参考手册，因为它承担判定功能。断言可以有固定字段。但这种固定字段只属于断言条目，不应污染所有章节。

所以我修正后的原则是：**章节采用适合其对象的文体。定义章用定义论述，流程章用过程论述，状态章用状态模型，断言章用可判定条目，附录用查阅表。**

## 五、断言仍然使用标题单位，但正文不应被断言体裁吞没

关于断言写法，我维持修正后的判断：断言作为标题单位存在，方便交叉引用。

但我要补充一条：断言不应该提前吞没整本书的表达。书的前半部分不是断言库。前半部分应该先让读者理解对象。只有当对象、流、投影稳定以后，断言才把这些内容转成可判定条目。

也就是说：

- Part 100 到 Part 300 是规约论述。
- Part 400 是符合性断言。
- Part 500 是治理与声明。

断言作为标题单位出现在 Part 400。其他章节可以引用断言，但不需要把自己写成断言。

断言标题的写法应自然。例如：

```asciidoc
[#camera-capture-state-visible.assertion, normative=must]
=== 摄像头采集状态可见
```

而不是：

```asciidoc
=== ASSERT-CAM-001
```

标题应给读者读，ID 给工具和交叉引用用。

## 六、关于小笔记中的三本文献坐标

你给的小笔记提到《如何阅读一本书》《金字塔原理》《风格的要素》。这些坐标我应当吸收，但不能把它们变成正文装饰。

### 1. 《如何阅读一本书》给结构审查坐标

Adler 和 Van Doren 强调统一性、结构、词义、命题、论证。对这本书的约束是：

- 全书必须能用一句话说明主题。
- 每个 part 必须能说明它如何支撑主题。
- 关键术语必须稳定。
- 每章要有明确命题，不写含糊感想。
- 论证要显示前提和推导，不只列结论。

这会影响我写作，但不一定需要在正文里大篇幅引用。

### 2. 《金字塔原理》给组织审查坐标

Minto 的价值在于结论先行、以上统下、分组清楚、逻辑递进。

对这本书来说，每个 part 的 partintro 应直接告诉读者本部结论：本部要建立什么对象、消除什么不确定性。每章第一段应给出该章的核心命题，不绕。

但我要避免把 MECE 当成机械分类。我们写的是书，不是咨询 slides。章节之间要有阅读流，不只是分类表。

### 3. 《风格的要素》给语言审查坐标

Strunk 和 White 的“省略废话”在这里非常重要。省略废话不是写短，而是删除不能降低读者不确定性的文字。一本 2 万 token 的书仍然可以没有废话，因为每一段都承担对象定义、论证、场景、判定或边界职责。

这也和你之前的《言语沟通》笔记一致：胡适的“须言之有物”、Grice 的量与关系准则、Shannon 的信噪比，都要求我不要写作者自我管理废话，不要写姿态词，不要把后台过程放进正文。

## 七、书中应出现什么样的语言

这本书的语言应当是清楚、稳定、具体、可审查的中文。

例如对象定义可以这样写：

> 实时全模态视频通话体验，是用户在同一会话中通过麦克风提供语音、通过摄像头或等价视觉输入提供当前视觉现场，并由 AI 以自然语音对语音内容、视觉现场和会话上下文作出连续回应的交互体验。

这句话是书中语言。它定义对象。

不应写成：

> 本章将介绍我们要做的体验，后续会继续完善。

这就是过程语言。

再例如视觉指代可以这样写：

> 视觉指代发生在用户的语言表达需要当前视觉现场才能确定所指对象时。用户说“这个”“刚才那个”“屏幕上这个错误”时，语音文本本身不足以确定对象；符合实现必须使用当前或可承接的视觉现场完成指代消解，或明确告知用户视觉现场不可用。

这就是降熵。它让“这个”不再靠心照不宣理解。

打断可以这样写：

> 打断不是停止扬声器播放的局部动作。打断改变当前回答的会话命运：被打断的回答不得继续作为用户可听输出推进，后续回答必须以用户的新输入为当前交互焦点。

这就是动态语义。

错误投影可以这样写：

> 当系统不能使用摄像头画面时，符合实现必须把限制投影给用户。沉默失败会让用户误以为 AI 正在看见现场，从而破坏实时全模态体验的公共契约。

这就是公共投影。

这些语言才适合进入书。我的写作清单不进入书。

## 八、这本书如何处理“完成”

你强调待确认书稿应当是完整的。我接受这个定义。

因此，书稿写到你面前时，不能是“这里以后补”“本章待写”“断言待补”“术语待补”的状态。可以存在你审阅后要求修改的地方，但不能存在我自己明知未完成还交给你的地方。

书稿完成应满足四个维度：

### 1. 主题完整

它完成了自己承诺的主题：定义并判定实时全模态视频通话体验。

### 2. 结构完整

从对象、体验、投影、断言到治理形成闭环。没有只开标题不展开的章节。

### 3. 论证完整

关键要求都有前提、解释和适用边界。不是只有“必须支持”这种孤立句子。

### 4. 查阅完整

读者能通过目录、稳定标题、交叉引用、术语表、附录和索引找到信息。

这四点没有达到，不叫待确认书稿，只叫未完成工作。未完成工作不应该交给你作为成果。

## 十、关于书中边界表达的正确方式

我之前说每章要写“本章边界”，容易变成模板化废话。正确方式是：只有当边界是理解该对象所必需时，才在正文自然表达。

例如《体验对象》章需要边界，因为读者很容易把它误认为语音助手、拍照问答、视频会议或数字人。这些相邻对象会造成高成本误读，所以需要说明。

但《摄像头采集状态》这样的章节不一定要写“本章不处理具体 UI 样式”。它只需要正面写：符合实现必须让用户观察到摄像头采集状态，具体视觉样式由实现决定。这样就自然表达了边界，不需要栏目化。

也就是说：边界不是固定栏目，而是必要时出现的论述功能。

## 十一、关于“规范性要求”的正确位置

我之前说每章都要有规范性要求，这也过硬。有些章节是解释性章节、术语章节、参考产品附录，不一定每章都有“必须”。强行要求每章都有规范性词，会导致滥用规范词。

正确方式是：

- 定义性章节负责定义对象和术语。
- 论证性章节负责解释为什么某个承诺属于对象。
- 规约性章节负责写必须、不得、应当、可以。
- 断言章节负责写黑盒场景。
- 附录章节负责查阅和模板。

规范性语言只在需要承担符合性约束的位置出现。不能为了看起来严格而到处写“必须”。

这也符合 RFC 2119 的精神：规范性词汇只在确有互操作、合规、风险意义时使用。

## 十二、关于“好书”的内化标准

我会把你给的小笔记内化成四条写作判据：

### 1. 统一性

每章必须支撑全书主题。不能出现离题技术介绍、供应商评测、作者过程说明。

### 2. 结构性

不写废话，不写姿态，不写自我安慰，不写“大家都懂”。长可以，但每段必须有信息功能。
