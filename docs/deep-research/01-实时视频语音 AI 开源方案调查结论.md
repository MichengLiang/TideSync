# 实时全模态视频通话 AI：开源技术方案调查与推荐

我已经按你给的目标做了一轮交叉调查：目标不是单纯“摄像头 + 麦克风 + 模型 API”，而是要接近 ChatGPT 高级语音、Gemini Live、豆包视频通话这种消费级体验：用户开摄像头和麦克风，AI 能实时听、实时看、用语音自然回应，支持打断、上下文连续、必要时接工具和联网信息。

下面是我的结论。先说最重要的判断：**目前没有一个完全开源、拿来就能一键复刻 ChatGPT / Gemini Live / 豆包视频通话的完整成品**。这些头部产品的模型、调度、端侧策略、音视频 QoS、记忆和安全策略大多是闭源系统。但现在开源生态已经足够拼出一个可用且工程上合理的版本，尤其你已经有了阿里云百炼的 `qwen3.5-omni-flash-realtime` 这个基座模型，难度大幅降低。真正要选的是：用哪一套实时音视频与 Agent 框架承接前后端媒体流、会话编排、客户端、监控、扩展能力。

我的推荐排序是：

1. **首选：LiveKit + 自定义 Qwen Realtime Adapter**。这是最接近生产级的路线，适合做 Web / 移动端“AI 视频通话”产品。
2. **第二选择：Pipecat + 自定义 Qwen Realtime Service**。适合快速拼多模型流水线、验证 Agent 行为、做服务端音视频/语音管线实验。
3. **强视觉场景可考虑：Stream Vision Agents**。适合健身私教、动作纠正、商品识别、现场导览、低延迟视觉处理这类“视频理解是核心”的场景。
4. **如果你未来想和 Agora 深度绑定：TEN Framework**。它定位也很准，但我会把它放在候选而不是第一落地方案。
5. **不建议从 mediasoup / Janus / Pion / aiortc 这些底层 WebRTC 组件直接重造产品**，除非你的目标是研究底层实时通信系统，而不是尽快做出 AI 视频通话体验。

## 一、先校准产品目标：闭源产品到底做了什么

### 1. ChatGPT 高级语音 + 视频

OpenAI 官方材料能确认两件事。第一，GPT-4o 这一代的方向就是原生多模态：文本、音频、图像、视频可以作为输入，输出可以是文本、音频、图像；OpenAI 公布过音频输入响应最快 232ms、平均 320ms 这一组指标。第二，ChatGPT Voice FAQ 明确提到移动端语音对话中可以按摄像头按钮共享设备视频，Release Notes 也记录了 advanced voice 中 real-time video、screen share、image upload 的 rollout。OpenAI Realtime API 官方文档也给了 WebRTC 接入方式，这说明 OpenAI 在开发者侧也把 WebRTC 当成浏览器实时语音 Agent 的标准入口。

需要注意：ChatGPT 产品本身闭源，官方公开 API 与 ChatGPT App 的内部产品栈不是一回事。我们能学到的是范式：**端侧实时媒体输入 + 原生实时模型 + WebRTC/低延迟音频 + 打断 + 上下文管理**，但无法直接复用 ChatGPT 的产品实现。

参考：
- https://openai.com/index/hello-gpt-4o/
- https://help.openai.com/en/articles/8400625-voice-mode-faq
- https://developers.openai.com/api/docs/guides/realtime-webrtc

### 2. Google Gemini Live / Project Astra

Google 官方资料同样能确认产品方向。Gemini Live 已经支持 camera 和 screen sharing；Google AI for Developers 的 Gemini Live API 文档写得更直接：Live API 面向低延迟、实时语音和视觉交互，处理连续音频、图像、文本流，输出即时的类人语音响应。Google Cloud 的 Agent Platform 文档还用“audio, video, or text”描述 Live API 的输入。

这对我们有两个启发：第一，消费级体验不是“拍一张图问答”，而是持续视觉上下文。第二，Google 自己也把这类能力产品化为 Live API，而不是要求开发者自己拆成 STT + VLM + TTS。

参考：

- https://ai.google.dev/gemini-api/docs/live-api

### 3. 豆包视频通话

豆包这边，公开信息比 OpenAI/Google 分散。字节 Seed 官方能确认的是豆包实时语音大模型：端到端语音对话、低时延、可打断、情绪承接。视频通话能力主要靠第三方实测和媒体报道交叉验证，例如每日经济新闻/证券时报转载报道提到豆包 App 上线视频通话，基于视觉推理模型并支持联网搜索，记者实测包括水果成熟度识别、持续记忆、逻辑推理等。

这里我的判断是：豆包视频通话产品存在，且确实进入了“用户摄像头画面 + 语音对话 + 视觉推理/搜索”的形态。但“0.5 秒延迟”“端侧如何抽帧调码率”等具体工程细节，公开可信来源不如 OpenAI/Google 官方文档完整，应该当作产品体验参考，而不是当作可精确复刻的技术规格。

## 二、你手上的 Qwen Realtime 基座模型，已经覆盖了最难的一层

阿里云官方文档明确写到：Qwen-Omni-Realtime 是实时音视频聊天模型，能理解流式音频与图像输入，例如从视频流实时抽取连续图像帧，并实时输出文本与音频。它支持 WebSocket 和 WebRTC 两种接入：WebSocket 适合服务端集成和快速接入；WebRTC 适合浏览器低延迟语音场景，音频走 UDP，内置回声消除和降噪。

更关键的是视频输入方式：

- WebSocket：客户端发送 `input_audio_buffer.append` 和 `input_image_buffer.append`，也就是音频流 + 图像帧。图像可以来自本地文件，也可以从视频流实时采集。
- WebRTC：建连时添加音频轨道和视频轨道，音频和视频通过 RTP 媒体通道传到服务端。官方说明 WebRTC 视频通过视频轨道发送画面帧，不需要 `input_image_buffer.append`。
- 官方 FAQ 还建议视频通话场景可以抽帧，频率建议 1 张/秒。这一点非常重要，因为它揭示了大多数“实时视觉对话”的真实工程做法：并不是每秒 30 帧全部进模型，而是实时通话媒体保持流畅，模型侧按策略采样关键帧/低频帧。

Qwen3.5-Omni-Realtime 还有几个对产品很重要的能力：联网搜索、Function Calling、语义打断、语音控制、服务端 VAD、长会话上限。阿里云文档显示会话最长 120 分钟；`qwen3.5-omni-flash-realtime` 的视频上下文最大轮次和最大时长也有明确限制，视频帧不是无限记忆，要做产品记忆仍需额外设计。

这意味着你不需要自己拼 STT + VLM + LLM + TTS 的完整链路来证明可行性。你现在有的是“端到端实时全模态模型 API”。剩下最重要的工程任务是：**如何把浏览器/移动端的摄像头麦克风、WebRTC 房间、后端 Agent、Qwen Realtime 会话、工具调用、前端体验稳定地连接起来**。

参考：
- https://help.aliyun.com/zh/model-studio/realtime
- https://help.aliyun.com/zh/model-studio/omni-realtime-python-sdk
- https://help.aliyun.com/zh/model-studio/server-events

## 三、生态里的开源方案分层

不要把“开源解决方案”理解成一个项目。这个系统至少有五层：

1. **客户端层**：Web / iOS / Android，负责摄像头、麦克风、扬声器、权限、屏幕共享、通话 UI、字幕、打断提示。
2. **实时媒体传输层**：WebRTC、SFU、TURN、弱网处理、音视频轨道、房间、设备切换。
3. **Agent 编排层**：管理用户会话、模型会话、上下文、工具调用、事件、VAD、打断、观测日志。
4. **模型接入层**：Qwen Realtime WebSocket/WebRTC，或者 OpenAI Realtime、Gemini Live、级联 STT-LLM-TTS。
5. **视觉增强层**：抽帧策略、YOLO/Roboflow/OpenCV/姿态估计、物体跟踪、场景状态缓存。

闭源产品把这五层打包成一个 App。开源世界里，成熟项目通常只覆盖其中两三层。真正合理的方案是选择一个强框架作为主干，然后补齐 Qwen Adapter。

## 四、候选方案评估

### 方案 A：LiveKit + 自定义 Qwen Realtime Adapter（首选）

LiveKit 是我最推荐的主干。它不是单纯的 AI 框架，而是一整套实时音视频基础设施：开源 WebRTC SFU、房间、token、客户端 SDK、服务端 SDK、SIP、录制、Ingress/Egress、Agent SDK。LiveKit 官方对 Agents 的定位是“realtime framework for voice, video, and physical AI agents”，支持把 Python/Node 程序作为实时参与者加入 LiveKit 房间，并通过 WebRTC 与前端通信。官方文档还明确说 LiveKit Agents 支持 voice、video、text，可处理实时输入并输出任意模态，支持 turn detection、interruptions、LLM orchestration、tools、多 Agent handoff、Kubernetes 部署。

我查了 GitHub 当前数据：`livekit/livekit` 约 19.2k stars，`livekit/agents` 约 10.9k stars，两个仓库截至 2026-06-12 仍有近期提交。许可证是 Apache-2.0。这说明它不是小众玩具，而是已经足够成熟的实时通信底座。

LiveKit 对你的价值：

- 前端 Web / iOS / Android / Flutter / React Native 生态完整。
- 摄像头、麦克风、屏幕共享、房间、弱网和音视频轨道不用你从零写。
- Agent 可以作为房间里的一个参与者，天然符合“用户与 AI 视频通话”的产品模型。
- 未来如果要多人房间、人工介入、客服、远程协助、录制、审计，LiveKit 的架构能承接。
- 你可以先用 LiveKit Cloud 快速跑通，也可以自托管 LiveKit Server。

短板也明确：LiveKit 官方目前没有现成的 `qwen3.5-omni-flash-realtime` Realtime 插件。LiveKit 有 OpenAI Realtime、Gemini 等集成，但 Qwen Realtime 需要自己写 adapter。这个 adapter 的工作包括：从 LiveKit 房间接收用户音频/视频轨道，按 Qwen 要求转成 WebSocket 音频流和图像帧，或直接研究 Qwen WebRTC 接入；把 Qwen 返回的音频流发布回房间；把字幕、转录、工具调用事件写回 data channel 或后端事件流。

我认为这是合理工作量，不是“造轮子”。因为 LiveKit 已经把最难的 WebRTC 产品基础设施做了，Qwen Adapter 只是模型接入层。

适合场景：正式产品、Web/移动端、视频通话 UI、后续多用户/客服/远程协助/录制/部署扩展。

参考：
- https://docs.livekit.io/agents/
- https://docs.livekit.io/agents/multimodality/vision/
- https://github.com/livekit/livekit

### 方案 B：Pipecat + 自定义 Qwen Realtime Service（第二推荐）

Pipecat 是 Daily 发起的开源实时语音/多模态 Agent 框架。官方定位很清楚：构建 voice and multimodal AI agents，可以 see、hear、speak。它的优点是 pipeline 抽象强，能把 transport、VAD、STT、LLM、TTS、frame processor、context aggregation 等环节串成低延迟流水线。Pipecat 文档里已经有 Gemini Live 服务，支持音频、视频、文本，多模态处理、实时流、VAD、Function Calling、上下文管理。它也有 Qwen LLM Service，但目前这个 Qwen 集成主要是 OpenAI-compatible 文本 LLM，不等于 Qwen Omni Realtime 音视频 API。

GitHub 当前数据：`pipecat-ai/pipecat` 约 12.8k stars，BSD-2-Clause，近期仍活跃。这说明 Pipecat 的社区活跃度和工程质量都不错。

Pipecat 对你的价值：

- 非常适合做“模型/管线实验台”：可以替换 OpenAI、Gemini、Qwen、Deepgram、Cartesia、ElevenLabs 等组件。
- 如果你未来要比较“Qwen 端到端”与“STT + LLM + TTS 级联”的延迟和成本，Pipecat 很方便。
- 它的 frame/pipeline 模型适合把图像帧、音频帧、转录、模型事件作为统一流处理。
- 服务端 Python 生态友好，和你当前 workbench 的 Python 主环境也匹配。

短板：它不像 LiveKit 那样本身就是完整 SFU/房间产品底座。Pipecat 可以接各种 transport，但如果目标是消费级视频通话 App，仍需要你选择 RTC provider 或前端连接方式。另一个短板同样是 Qwen Realtime 需要自定义 service，现成 Qwen LLM 不能直接覆盖音视频实时通话。

适合场景：原型验证、模型管线比较、快速做 Agent 行为实验、服务端多模型编排、未来可能混合 STT/LLM/TTS。

参考：
- https://docs.pipecat.ai/overview/introduction
- https://docs.pipecat.ai/api-reference/server/services/s2s/gemini-live
- https://docs.pipecat.ai/api-reference/server/services/llm/qwen

### 方案 C：Stream Vision Agents（强视觉场景候选）

Vision Agents 是 Stream 团队新推的开源 Python 框架，定位非常贴合“实时视频 AI Agent”。官方说它是 open-source Python framework for voice and video AI agents，支持实时 WebRTC、25+ integrations、native tool calling、pluggable vision pipelines。它的示例代码直接就是 Agent + Gemini Realtime + YOLOPoseProcessor 这种模式。Stream 官方博客还说 Vision Agents 支持 OpenAI Realtime 和 Gemini Live，并能处理 tracks、video subscriptions、response type 转换等复杂性。

GitHub 当前数据：`GetStream/Vision-Agents` 约 7.9k stars，Apache-2.0，近期活跃。它比 LiveKit/Pipecat 年轻，但增长快，而且主题非常贴合“视频 Agent”。

Vision Agents 对你的价值：

- 它不是泛泛的 voice agent，而是偏“实时视觉 AI 应用”。
- 内置思路就是把 YOLO、Roboflow、pose detection、Gemini/OpenAI Realtime 等组合起来。
- 如果你做健身私教、动作纠正、球类教练、商品识别、现场导览、监控告警，它的处理器机制更贴近业务。
- 它能避免你把所有视觉任务都丢给 Qwen。比如姿态检测、物体框、计数、OCR/检测可以本地或边缘先做，Qwen 负责语言理解、推理和语音回应。

短板：它强绑定 Stream 的 realtime transport 体验，虽然官方说也能与其他 WebRTC 基础设施配合，但最顺的路线显然是 Stream 账号和 Stream edge。另一个问题是它更年轻，生态稳定性和生产部署经验还需要谨慎评估。接 Qwen Realtime 也同样需要自定义 integration。

适合场景：视频理解是核心卖点，而不只是“AI 偶尔看一眼摄像头”。例如健身教练、桌面操作指导、维修指导、商品验真、体育评论、安防辅助。

参考：
- https://visionagents.ai/
- https://getstream.io/vision-agents/
- https://github.com/GetStream/Vision-Agents
- https://getstream.io/blog/vision-agents-by-stream/

### 方案 D：TEN Framework（Agora 生态候选）

TEN 是另一个值得关注的实时多模态 conversational AI 框架。官方文档写明它是 open-source framework for real-time multimodal conversational AI，支持模块化、扩展、低延迟、多语言扩展。TEN Agent Examples 明确提到 Gemini 2.0 Live 和 OpenAI Realtime API，架构上编排 ASR、LLM、TTS、RTC 组件，支持 full-duplex conversation 和自然打断。Agora 页面也把 TEN 描述为可创建能 see、hear、speak 的实时多模态 AI，并且由 Agora 和社区支持。

GitHub 当前数据：`TEN-framework/ten-framework` 约 10.7k stars，近期活跃。但我用 GitHub API 查到 license 字段是 `NOASSERTION`，正式商用前需要人工核对仓库许可证和各组件许可证。

TEN 的价值：

- 它强调模块化扩展，适合做复杂 Agent 运行时。
- 与 Agora 生态有天然关联，Agora 在实时音视频商业化和全球网络上经验很强。
- 它自带 VAD、Turn Detection 等组件方向，适合 full-duplex 语音体验。

短板：从公开资料看，TEN 的文档和生态成熟度不如 LiveKit/Pipecat 清晰。它适合进入 PoC 候选，但我不建议把第一版产品押在 TEN 上，除非你已经决定使用 Agora 作为 RTC 基础设施，或者你明确喜欢它的 extension 架构。

参考：
- https://theten.ai/docs
- https://theten.ai/docs/ten_agent_examples/overview
- https://prod.agora.io/en/open-source/ten-framework

### 方案 E：直接用 Qwen 官方 WebRTC/WebSocket 示例 + 自己写前端后端

这是最快验证模型的办法。阿里云官方文档本身已经有 Python SDK、WebSocket、WebRTC、函数计算模板。你可以先不引入 LiveKit/Pipecat，直接做一个最小 Web App：浏览器拿摄像头麦克风，服务端连接 Qwen Realtime，音频流和抽帧图像发给 Qwen，Qwen 音频返回给浏览器播放。

优点：快、依赖少、最能验证 Qwen 模型体验、成本最低。缺点：一旦要产品化，你会很快遇到房间管理、弱网、设备切换、回声消除、断线恢复、监控、权限、安全、TURN、移动端兼容等问题。也就是说，这条路线适合“第一周验证”，不适合“长期产品主干”。

我建议把它作为第 0 阶段，而不是最终架构。

## 五、底层 WebRTC 轮子：为什么不建议从它们开始

你会看到很多底层项目：mediasoup、Janus、Pion、aiortc。

- **mediasoup**：强大的 Node/C++ WebRTC SFU，适合团队有 RTC 专家、想高度自定义房间和媒体路由。
- **Janus Gateway**：老牌 C WebRTC server，插件体系成熟，但产品层开发体验不如 LiveKit 现代。
- **Pion WebRTC**：Go 语言 WebRTC 实现，LiveKit server 也基于 Pion。适合写底层媒体服务，不适合直接当应用框架。
- **aiortc**：Python WebRTC/ORTC 实现，适合研究、单机 demo、服务端点对点 WebRTC。阿里云文档的 WebRTC Python 示例也能看到类似思路。但它不提供完整产品级 SFU/房间体系。

这些都不是坏东西，但它们解决的是底层通信问题，不是 AI 视频通话产品问题。你的目标是搭出 ChatGPT / Gemini Live / 豆包式体验，所以不应该一开始就掉到底层媒体协议里。除非你有明确的定制 RTC 需求，否则选 LiveKit 这类完整实时栈会更稳。

参考：
- https://github.com/livekit/livekit
- https://github.com/pion/webrtc
- https://github.com/aiortc/aiortc

## 六、关于“全开源本地模型”的现实判断

你现在用的是阿里云百炼 Qwen3.5-Omni-Flash-Realtime，是云 API。问题是：能不能完全开源、本地自托管出同样体验？

Qwen3-Omni 开源仓库确实存在，官方说明它是端到端 multilingual omni-modal foundation model，能处理文本、图像、音频、视频，并输出实时流式文本和自然语音。vLLM-Omni 也已经有 Qwen3-Omni serving 文档。但是从第三方技术教程的延迟评测看，现实还没那么理想：DashScope Realtime API 可以做到约 702ms 平均 TTFA；vLLM Thinker 自托管可以很快做 audio-to-text/text generation，但没有音频输出；Transformers 全量 Thinker+Talker 自托管音频输出非常慢，不适合实时。该教程的结论是：优化过的 Talker 实时音频生成目前主要在 DashScope cloud API 中可用，完全自托管的端到端实时语音还不成熟。

所以我建议：**第一版不要追求全本地开源模型**。用 DashScope Realtime 做主模型，把系统架构设计成 provider-adapter 模式，将来如果 vLLM-Omni 的 Talker serving 真正成熟，再替换模型层。这样不会把产品进度卡死在本地推理性能上。

参考：
- https://github.com/QwenLM/Qwen3-Omni
- https://docs.vllm.ai/projects/vllm-omni/en/latest/user_guide/examples/online_serving/qwen3_omni/
- https://arxiv.org/html/2603.05413v2

## 七、我推荐的落地架构

### 第一阶段：最小可用原型

目标：验证 Qwen3.5-Omni-Flash-Realtime 的真实交互体验，而不是先搭大系统。

做法：

- Web 前端打开摄像头和麦克风。
- 音频以 16k PCM 或 SDK 要求格式送 Qwen。
- 视频侧按 1 fps 抽帧，先走 WebSocket `input_image_buffer.append`。
- Qwen 输出音频流，前端播放；同时显示转录和回复文本。
- 打断先用 Qwen 服务端 VAD/semantic VAD 能力，前端检测到用户说话时停止当前播放。
- 记录每轮 first text delay、first audio delay、端到端响应时间、丢帧/断线情况。

这个阶段可以直接参考阿里云官方 SDK 示例，不必先引入 LiveKit。验收标准是：在真实网络下，是否能做到“可连续对话、可看画面、能打断、延迟可接受”。

### 第二阶段：LiveKit 产品化骨架

目标：把原型升级为“像通话产品”的结构。

建议架构：

- Browser / Mobile App：LiveKit Client SDK 发布 mic/camera/screen tracks。
- LiveKit Server / Cloud：负责房间、音视频轨道、弱网、TURN、token、参与者管理。
- Qwen Agent Service：作为 LiveKit room participant 加入房间。
- Adapter：订阅用户音频轨道和视频轨道；音频转 Qwen 输入；视频按策略抽帧或尝试 Qwen WebRTC 视频轨道；接收 Qwen 音频输出后发布为 agent audio track；文本/字幕/状态通过 data channel 发给前端。
- Tool Service：天气、搜索、业务 API、RAG、地理位置、商品库等工具。
- Observability：记录会话 id、延迟、首包时间、VAD 事件、模型错误、工具调用、用户反馈。

这就是我认为最合理的主线。

### 第三阶段：视觉增强

Qwen 可以理解画面，但产品不能把所有视觉任务都丢给大模型。要想接近豆包/Gemini Live 的“稳定可用”，需要加视觉状态层：

- 抽帧策略：默认 1 fps；用户问“看这里/这个是什么”时临时提高到 2-5 fps；静止画面降频。
- 关键帧缓存：保存最近 N 秒的帧摘要，而不是保存所有图像。
- 视觉状态表：桌面上有哪些物体、用户正在指什么、上一次提到的物体是什么。
- 轻量 CV：YOLO/Roboflow/OpenCV/姿态估计先做结构化检测，再让 Qwen 解释。
- 场景触发：健身动作不标准、物体进入画面、屏幕出现错误码等，可以由本地视觉处理器触发模型回应。

如果你的产品偏视觉教练/动作纠正，这一层可以考虑引入 Vision Agents。如果只是普通 AI 视频通话，LiveKit + 自己的 frame sampler 足够。

## 八、推荐路线图

我建议按四步走：

1. **3-5 天：Qwen 官方 Demo 级原型**
   - 目标是测模型，不测框架。
   - 做浏览器摄像头/麦克风、抽帧、Qwen 回音频、字幕日志。
   - 产出真实延迟数据和问题清单。

2. **1-2 周：LiveKit 房间化原型**
   - 前端用 LiveKit SDK。
   - 后端 Agent 加入房间。
   - Qwen Adapter 跑通音频输入、视频抽帧、音频输出。
   - 支持打断、断线重连、设备切换、基础状态展示。

3. **2-4 周：产品体验打磨**
   - 加字幕、说话状态、AI 正在看/正在听提示。
   - 增加工具调用和联网搜索策略。
   - 加视觉上下文缓存。
   - 做成本控制：抽帧频率、音频时长、静音过滤、会话时长限制。
   - 做隐私提示：摄像头画面如何使用、是否存储、如何关闭。

4. **后续：场景化增强**
   - 如果做健身/维修/商品识别，接 Vision Agents 或自写 CV processors。
   - 如果做客服/会议/远程协助，接 LiveKit 录制、人工接管、CRM/RAG。
   - 如果做移动 App，优先用 LiveKit iOS/Android SDK，而不是 WebView 硬撑。

## 九、最终建议

如果你问“现在最主流、成熟、可落地的开源技术方案是什么”，我的答案是：

**LiveKit 做实时音视频与产品骨架，Qwen3.5-Omni-Flash-Realtime 做实时全模态模型，自写一层 Qwen Realtime Adapter；视觉场景重时再引入 Vision Agents 或自研轻量视觉处理器；Pipecat 保留为模型管线实验和备选 Agent 编排框架。**

这条路线的优点是稳：LiveKit 解决通话产品最麻烦的 WebRTC 和多端 SDK，Qwen 解决最难的端到端多模态理解和语音输出，你只在中间写必要胶水层和业务逻辑。它不会被某个年轻框架绑死，也不会让你从 mediasoup/Pion 这种底层通信开始重造整个产品。

我不建议的路线是：

- 不建议第一版追求全本地 Qwen3-Omni 自托管实时语音输出。现在云端 DashScope Realtime 明显更现实。
- 不建议第一版直接基于 mediasoup/Janus/Pion 从零写产品层，除非你团队本身就是 RTC 团队。
- 不建议把普通 Qwen OpenAI-compatible Chat Completions 当成 Realtime Omni API 使用，它只能覆盖文本 LLM，不等于实时音视频模型。
- 不建议只做“每隔几秒截图 + 普通 VLM + TTS”的假实时方案，体验上会明显输给 ChatGPT/Gemini/豆包式通话。

工程上最关键的三个风险是：

1. **Qwen Adapter 的实时音频格式和打断处理**：必须认真处理音频采样率、播放缓冲、用户说话时清空输出、服务端 VAD 事件。
2. **视频抽帧和上下文成本**：视觉不是帧率越高越好，过高会烧 token、增加延迟、污染上下文。默认 1 fps 是合理起点。
3. **产品侧状态管理**：用户需要知道 AI 是否正在听、正在看、正在回答、是否被打断；否则技术可用但体验会混乱。

如果下一步进入实现，我建议先做一个 `experiments/qwen-realtime-video-call/` 小项目：第一个里程碑只接 DashScope WebSocket + 浏览器抽帧；第二个里程碑再接 LiveKit。这样路径最短，也最容易用真实体验决定后面要不要上完整框架。
