### 3.3 metrics endpoint

`api.py` 第 194-222 行已经给出 metrics endpoint。`models.py` 第 30-43 行定义 `GetAgentSessionMetricsResponse`，字段为：

- `session_id`
- `call_id`

- `session_started_at`

前端边界结论：TideSync 前端可以稳定展示 metrics，但 metrics 字段是 `dict[str, int | float | None]`，需要以动态指标表呈现。它不是强 schema 的 Qwen event stream。React Query 负责轮询与错误状态，UI 负责显示生成时间、session id 和数值。

### 3.4 custom call event 能力

`stream_edge_transport.py` 第 485-500 行定义 `send_custom_event(data: dict)`：

- 注释明确 custom events delivered to clients subscribed to the call via `call.on("custom", callback)`
- payload 限制 5KB
- `_call is None` 时抛 `RuntimeError`
- 最终调用 `self._call.send_call_event(user_id=self._agent_user_id, custom=data)`

GetStream 官方 Events 文档也说明 call events 可由 `client.on(...)`、join/watch/query watch 接收，并给出 `StreamVideoClient` 订阅所有事件的形式。

前端边界结论：TideSync 自研前端可以通过 `@stream-io/video-client` 或 React SDK 暴露的 call 对象订阅 `custom` 事件，用于显示 agent-side 状态、Qwen snapshots、tool/search/usage 状态。该能力来自 GetStream call event，不是 Vision Agents 浏览器 SDK。

### 3.5 不存在“Vision Agents 浏览器 SDK”

本地 fork 代码显示 Vision Agents 的浏览器相关面由 Runner HTTP API 与 GetStream edge 间接提供。`agents-core/vision_agents/core/edge/events.py` 第 11-60 行只定义后端内部 PluginBaseEvent：audio received、track added/removed、call ended、participant joined/left。GetStream plugin 是 Python 端 `AsyncStream`、`rtc.join`、`aiortc`、`getstream.video.rtc`。Qwen plugin 是 Python 端 WebSocket client。没有本地源码证据显示存在 `vision-agents` 的浏览器 npm SDK、浏览器 Qwen SDK 或前端可直接 import 的 Vision Agents JS 包。

结论：TideSync 前端架构必须表述为“React + GetStream browser SDK + Runner HTTP API + GetStream custom events”，不能表述为“Vision Agents 浏览器 SDK”。

## 4. Qwen 本地文档中的前端相关能力分类

### 4.1 WebSocket 与 WebRTC

`docs/Qwen3.5-Omni-Flash-Realtime-API.md` 第 11 行明确：Qwen-Omni-Realtime 支持 WebSocket 和 WebRTC。WebSocket 适合服务端集成和快速接入；WebRTC 适合浏览器端、低延迟语音场景，音频通过 UDP 直接传输，内置回声消除和降噪。

第 286-308 行进一步区分输入：

- WebSocket：第 292 行，客户端用 `input_audio_buffer.append` 和 `input_image_buffer.append` 发送 Base64 音频与图片。
- WebSocket 手动模式：第 296 行，禁用 VAD 时客户端必须发送 `input_audio_buffer.commit`。
- WebRTC：第 300-304 行，建连时的音频轨道和视频轨道通过 RTP 自动传输；音频无需 `input_audio_buffer.append`；图片通过视频轨道发送，不支持 `input_image_buffer.append`。
- 第 307 行：WebRTC 仅支持服务端 VAD 模式，不支持手动模式。

TideSync 当前路线分类：

- 前端配置面：GetStream call join、浏览器麦克风/摄像头权限、mute/camera toggle、call participant/audio/video UI、custom event 订阅。
- 后端 agent 配置面：Qwen WebSocket base URL、model、voice、modalities、input/output audio format、turn_detection、tools、search、input audio transcription。
- 相邻对象：Qwen 原生 WebRTC 直连浏览器。它属于 Qwen 官方能力，但 TideSync 当前通过 GetStream + Vision Agents Python edge 接入，不直接让浏览器连 Qwen WebRTC。

### 4.2 VAD / Manual

第 2984-2986 行明确 VAD 模式：`session.update.session.turn_detection.type` 为 `"server_vad"` 或 `"semantic_vad"`；WebSocket 和 WebRTC 均支持 VAD，服务端事件一致，区别是音频/图片传输方式。第 3009-3023 行明确 Manual 模式：`session.turn_detection` 设为 `null`，客户端显式发送 `input_audio_buffer.commit` 与 `response.create`。

本地 Qwen adapter `qwen_realtime.py` 第 425-443 行构造参数包含：

- `turn_detection: "server_vad" | "semantic_vad" | None = "server_vad"`
- `vad_threshold`
- `vad_prefix_padding_ms`
- `vad_silence_duration_ms`
- `tools`
- `enable_search`
- `search_options`

第 542-552 行 `_build_turn_detection_config()` 将 `None` 映射为无 turn_detection，将 `"server_vad"` / `"semantic_vad"` 映射为 Qwen config。

分类：

- 前端配置面：调试 UI 可以提供 VAD 类型选择、threshold、prefix padding、silence duration、Manual 开关，但这些只是配置输入。
- 后端 agent 配置面：实际 `session.update` 由 Qwen adapter 构造并发送。Manual 模式下 commit/create_response 由后端 agent 对 Qwen 调用，不由浏览器直接对 Qwen 调用。
- 相邻对象：Qwen 原生 WebRTC 不支持 Manual，因此若 TideSync 后续采用 Qwen 原生 WebRTC，Manual 不能作为浏览器直连模式。

### 4.3 `session.update`

Qwen 文档多处以 `session.update` 作为会话配置入口：VAD 第 2984 行，Manual 第 3011 行，联网搜索第 3044-3051 行，工具调用文档第 7 行。Qwen adapter 本地实现 `_build_session_config()` 第 520-537 行构造：

- `modalities`: `["text", "audio"]`
- `voice`
- `instructions`
- `input_audio_format`: `"pcm"`
- `output_audio_format`: `"pcm"`
- `input_audio_transcription`: `{"model": self._audio_transcription_model}`
- `turn_detection`
- `tools`
- `enable_search`
- `search_options`

分类：

- 前端配置面：表单承载这些字段的可编辑值、校验和展示。
- 后端 agent配置面：字段写入 Qwen session，并处理 Qwen 错误。
- 相邻对象：DashScope Python/Java SDK 示例调用 `update_session`，不是 TideSync React 前端直接依赖。

### 4.4 音频与视频输入

Qwen 文档第 286-308 行定义音频必需、图片可选；WebSocket 通过 append audio/image；WebRTC 通过 RTP 音视频轨道。文档第 384-386 行说明视频以抽帧方式输入，推荐 1 帧/秒；第 390-394 行列出 qwen3.5 flash/plus 的音频和视频轮次/时长限制。

本地 Qwen adapter 第 422-443 行包含 `include_video`、`fps`、`video_width`、`video_height`；第 583-589 行 `simple_audio_response()` 发送 PCM 音频并标记 audio appended；代码中还存在 `VideoForwarder`、JPEG frame 处理。

分类：

- 前端配置面：摄像头开关、call 内视频 track 是否发布、视频预览、屏幕共享显示、调试面展示当前 FPS 配置。
- 后端 agent 配置面：Qwen WebSocket image append、抽帧、JPEG 编码、图像时序控制、每轮视频发送权限。
- 相邻对象：Qwen 原生 WebRTC 视频轨道传输；TideSync 当前用 GetStream call，浏览器视频先进入 Stream，再由 Vision Agents edge 订阅和转发到 Qwen。

### 4.5 response events、usage、search/tools

Qwen 文档第 313-336 行定义响应：

- WebSocket 文本：`response.text.delta` / `response.text.done`
- WebSocket 文本+音频：`response.audio_transcript.delta` / `.done`，音频 `response.audio.delta` / `.done`
- WebRTC 文本：DataChannel
- WebRTC 音频：RTP 轨道，不用 `response.audio.delta`

第 2998-3007 行列出 VAD 生命周期：`response.created`、`conversation.item.created`、`response.audio_transcript.delta`、`response.done`，以及 `response.output_item.added`、`response.content_part.added`、`response.audio.done`、用户语音转写 delta/completed 等。

联网搜索：第 3034-3056 行说明 `enable_search` 默认关闭，通过 `session.update` 启用，`response.done.usage.plugins.search` 返回 search count 和 strategy。第 380 行说明联网搜索和工具调用不兼容。

工具调用文档第 7-13 行说明通过 `session.update` 传工具定义；第 13-24 行服务端返回 `response.function_call_arguments.done`；第 32-45 行客户端通过 `conversation.item.create` 回传 `function_call_output`，再发送 `response.create`；第 50 行说明不支持 `tool_choice` 和 `parallel_tool_calls`。

本地 adapter 对应事实：`qwen_realtime.py` 第 84-105 行定义 Usage/Search/Tool 状态；第 284-367 行定义 tool call state；第 508-537 行构造 tools/search config，并在第 511-518 行禁止 tools 与 search 同时启用。

分类：

- 前端配置面：显示 response lifecycle、usage、search usage、tool call state、错误态、transcript、音频输出状态；提供 search/tools 开关和 tool schema 输入。
- 后端 agent 配置面：实际执行工具、处理 function call delta/done、回传 tool output、触发最终 response、解析 usage、处理 search/tools 互斥。
- 相邻对象：Qwen 原生工具调用 WebSocket demo 中的本地函数执行，不能直接搬到浏览器执行生产工具。TideSync 调试前端可以显示和注入测试 schema，但工具执行边界在后端 agent。

### 4.6 音色、声音复刻、语音控制

文档第 359-373 行说明：

- 支持语音指令控制声音大小、语速和情绪。
- 支持 55 种音色。
- `qwen3.5-omni-plus-realtime` 和 `qwen3.5-omni-flash-realtime` 支持声音复刻，可使用自定义音色实时对话。

分类：

- 前端配置面：voice 下拉、voice id 输入、自定义音色 id 显示、语速/情绪指令提示作为会话指令的一部分。
- 后端 agent 配置面：`voice` 字段写入 Qwen session；声音复刻生成出的自定义音色 ID 作为 voice 使用。

- interrupt 呈现：Qwen `input_audio_buffer.speech_started` 对应的打断状态应在前端有明确反馈。
- timer：用于显示 session 持续时间和 Qwen 120 分钟上限。

### 5.2 不可照搬的架构原因

`server.py` 事实：

- 第 1-8 行使用 FastAPI WebSocket、HTMLResponse、`OmniRealtimeClient`。
- 第 67-90 行创建 Qwen client，模型写为 `qwen-omni-turbo-realtime`，turn detection 固定 `SERVER_VAD`。
- 第 115-126 行直接从本地 `static/index.html` 返回页面。
- 第 128-260 行 `/ws` WebSocket 接收浏览器二进制音频/视频，转发到 Qwen WebSocket；音频事件手动构造成 `input_audio_buffer.append`；视频进入 queue 后 `append_image`。

不可照搬原因：

- 该 zipball 前端是“浏览器 WebSocket 到自带 FastAPI，再到 Qwen WebSocket”的架构。TideSync 的目标是“浏览器加入 GetStream call，Vision Agents Python edge 加入同一个 call，再由 Qwen adapter 与 Qwen Realtime 通信”。
- zipball 前端自己采集麦克风 PCM 和 JPEG 并发送到 `/ws`。TideSync 浏览器端应通过 GetStream SDK 发布音视频轨道，Vision Agents edge 从 Stream call 接收 audio/video，再转发到 Qwen。
- zipball 前端不使用 `@stream-io/video-react-sdk`、`@stream-io/video-client`、Runner HTTP API 或 GetStream custom call event。
- zipball server 把 Qwen API key 留在后端，方向正确；但它的 session/control/event 协议是临时私有 WebSocket，不等同于 Vision Agents Runner API 和 GetStream call event。
- zipball UI 是单 HTML/CSS/JS 文件，状态与 DOM 操作耦合。TideSync 自研前端需要 React 组件、类型化 event model、表单 schema、服务端状态缓存和 Playwright 验证。

结论：借鉴交互对象和状态呈现，不搬运通信架构与单文件 DOM 实现。

## 7. 确定设计约束清单

1. TideSync 前端通过 GetStream 浏览器 SDK 加入 call；不直接引入不存在的 Vision Agents 浏览器 SDK。
2. React 通话 UI 以 `@stream-io/video-react-sdk` 为主；底层 call event/custom event 以 `@stream-io/video-client` 的事件模型为事实依据。
3. Chat/message 能力只使用 `stream-chat` 与 `stream-chat-react`；不使用未核实的包名。
4. Runner HTTP API 只承担 agent session 生命周期、health/ready、metrics；Qwen 会话参数不由 Runner HTTP API 自动暴露。
5. GetStream token 获取必须由 TideSync 后端或受控 token endpoint 提供；`open_demo()` 中的 GetStream 官方 demo URL 只是现有调试逻辑证据。
6. 前端可以启动/关闭 Runner session、查看 session、轮询 metrics；不能把 metrics 当作 Qwen 原始事件流。
7. 前端可以订阅 GetStream custom call event；payload 需按 5KB 限制设计。
8. Qwen `session.update` 字段属于后端 agent 写入 Qwen 的配置；前端只提供受控配置输入和状态展示。
9. WebSocket Qwen 能力与 WebRTC Qwen 能力必须分开标注；TideSync 当前链路是 GetStream WebRTC + Vision Agents Python + Qwen WebSocket。
10. Manual 模式只对 WebSocket 接入成立；Qwen 原生 WebRTC 不支持 Manual。
11. 联网搜索与工具调用互斥；前端表单必须体现互斥约束，后端 adapter 已在本地 fork 中执行互斥校验。
12. 工具调用执行边界在后端 agent；前端显示 tool schema、调用状态、arguments、output、error，不承担真实业务工具执行。
13. 声音复刻不是 Realtime 会话内联创建能力；前端只接受已创建 voice 标识。
14. 音频输出、打断、stale audio、transcript final/delta、usage/search usage 都应作为独立调试状态显示。
15. 官方 zipball 的 voice selector、状态指示器、本地视频预览、mic/camera/stop/reconnect、history、timer 可作为交互对象参考；其 WebSocket 中继架构和单 HTML 实现不进入 TideSync 架构。
16. Playwright 覆盖浏览器权限、布局、连接状态、事件面板、表单验证和核心调试动作；Biome 覆盖 TS/React 代码风格。
17. UI 组件采用 Radix 基元保证 keyboard/focus/ARIA；图标使用 lucide；toast 使用 sonner；服务端状态使用 React Query；表单使用 React Hook Form + Zod。
18. 所有 Qwen、Runner、GetStream payload 在 TypeScript 中建模，避免调试面板依赖无结构对象。
